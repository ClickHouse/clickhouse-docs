Managed Postgres поддерживает создание изолированных веток базы данных с помощью восстановления на определённый момент времени (PITR).

Ветка — это полностью независимое развертывание PostgreSQL, созданное из состояния существующей базы данных на определённый момент времени. Ветки можно использовать для разработки, промежуточного хранилища, тестирования, отладки, проверки данных или сценариев восстановления — не затрагивая исходную базу данных.

В отличие от реализаций copy-on-write, которые используют общее хранилище с основной базой данных, ветки Managed Postgres восстанавливаются из резервных копий и работают как независимые развертывания PostgreSQL.

## Как работает ветвление \{#branching\}

Создание веток основано на той же инфраструктуре резервного копирования и восстановления, которая используется для [восстановления на определённый момент времени (PITR)](/cloud/managed-postgres/backup-and-restore).

Когда вы создаёте ветку, Managed Postgres восстанавливает базовую резервную копию из объектного хранилища, применяет необходимые сегменты WAL, чтобы достичь заданной точки восстановления, и разворачивает новый экземпляр PostgreSQL из восстановленного состояния. После завершения восстановления ветка работает независимо от исходной базы данных.

Получившаяся ветка — это полная копия исходной базы данных на выбранный момент времени.

## Типичные сценарии использования \{#common-use-cases\}

### Разработка и тестирование \{#dev-and-testing\}

Создайте ветку от рабочей или базы данных промежуточного хранилища, чтобы проверить изменения в приложении, миграции или новые возможности на реалистичных данных.

### Среды промежуточного хранилища \{#staging-environments\}

Поддерживайте среду промежуточного хранилища, максимально приближенную к production, не затрагивая рабочие нагрузки production.

### Проверка данных \{#date-validation\}

Проверяйте изменения схемы, стратегии индексирования и оптимизацию запросов перед развертыванием в рабочей среде.

### Восстановление и анализ \{#recovery-and-investigation\}

Восстановите базу данных на определённый момент времени для устранения неполадок, аудита или проверки поведения приложения.

## Размер веток \{#branch-sizing\}

Ветки — это независимые развертывания PostgreSQL, и их размер можно настраивать отдельно от исходной базы данных.

Например, производственное развертывание может работать на более крупной конфигурации, а ветка для разработки или промежуточного хранилища — использовать меньший вычислительный профиль, чтобы снизить затраты. Это позволяет командам создавать временные окружения, не выделяя им те же вычислительные ресурсы, что и для продакшена.

## Время создания ветки \{#branch-creation-time\}

Поскольку Managed Postgres использует хранилище PostgreSQL на базе NVMe, ветки восстанавливаются из резервных копий, а не создаются с помощью механизмов copy-on-write на уровне хранилища. Поэтому создание ветки не происходит мгновенно.

Обычно создание ветки занимает от нескольких минут до нескольких десятков минут в зависимости от:

* Размера базы данных
* Размера резервной копии
* Точки восстановления
* Объема WAL, который необходимо воспроизвести
* Общей конфигурации кластера

В большинстве развертываний ветки становятся доступны в течение нескольких минут. Для крупных баз данных может потребоваться больше времени.

Если время создания ветки становится узким местом в вашем рабочем процессе, свяжитесь с командой ClickHouse. Во многих случаях производительность восстановления веток можно оптимизировать с учетом характеристик рабочей нагрузки и требований к восстановлению.

## Ветки vs локальная разработка \{#branches-v-local-dev\}

Часто возникает вопрос: должен ли каждый разработчик использовать ветку production в качестве своей среды разработки.

Хотя ветки полезны для тестирования, валидации и процессов промежуточного хранилища, в целом это не лучший подход для повседневной разработки приложений. Каждая ветка — это отдельное развертывание PostgreSQL, которое нужно восстанавливать из резервных копий и поддерживать отдельно. Создание большого числа веток может увеличить затраты на инфраструктуру и усложнить эксплуатацию.

Для большинства организаций мы рекомендуем:

* Использовать ветки PostgreSQL для промежуточного хранилища, тестирования, отладки и валидации.
* Использовать локальные среды PostgreSQL для повседневной разработки.
* Генерировать синтетические наборы данных для разработки или использовать обезличенные наборы данных, где это уместно.
* Избегать повседневной разработки напрямую в ветках, созданных на основе production.

Такой подход снижает нагрузку на production-системы, ускоряет разработку и помогает обеспечить надлежащую защиту production-данных.

Руководство по созданию локальных сред разработки PostgreSQL с помощью Docker см. в разделе [Локальные среды разработки](/cloud/managed-postgres/local-development).

## Рекомендуемый порядок работы \{#recommended-workflow\}

Обычно процесс выглядит так:

```text
Production Database
        │
        ├─────────────► Branch
        │                  │
        │                  ├── Staging
        │                  ├── Validation
        │                  ├── Migration Testing
        │                  └── Incident Investigation
        │
        └─────────────► Local Development
                               │
                               ├── Docker PostgreSQL
                               ├── Application Migrations
                               └── Synthetic Test Data
```

Ветки лучше всего подходят для сред, которым нужна копия базы данных, приближенная к production. Для повседневной разработки локальные среды PostgreSQL обычно обеспечивают более быстрый, менее затратный и лучше масштабируемый рабочий процесс.
