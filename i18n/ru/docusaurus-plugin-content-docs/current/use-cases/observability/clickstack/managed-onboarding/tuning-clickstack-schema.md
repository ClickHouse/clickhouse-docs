Если вы уже некоторое время используете ClickStack, то, вероятно, заметили, что схема по умолчанию справляется с большинством задач обсервабилити без каких-либо изменений. Эта страница — для случаев, когда этого уже недостаточно: задержка запросов начинает расти или ваши шаблоны доступа к данным уже не соответствуют настройкам по умолчанию.

Четыре оптимизации охватывают большую часть того, что действительно помогает на практике. Они перечислены примерно в порядке возрастания трудоёмкости. Первые две — это локальные изменения `ALTER TABLE`, которые можно внедрять постепенно. Третья даёт эффект, когда одна и та же агрегация снова и снова выполняется на панели мониторинга. Четвёртая требует миграции таблицы, поэтому она самая сложная.

Приведённые ниже сводки намеренно кратки. Обоснование каждого изменения, результаты бенчмарков и инструкции по их применению к существующим данным см. в разделе [Настройка производительности](/use-cases/observability/clickstack/performance_tuning).

  ## Материализуйте часто используемые атрибуты \{#materialize-attributes\}

  Фильтрация по `LogAttributes['service.version']` заставляет ClickHouse загружать и декодировать весь Map `LogAttributes` для каждой проверяемой строки. Вынесите этот атрибут в `MATERIALIZED`-столбец, и тот же фильтр превратится в чтение столбца, что обычно на порядок быстрее. Как только столбец появится, ClickStack автоматически перепишет фильтр, поэтому сохранённые поиски и панели мониторинга продолжат работать без изменений.

  Выбирайте только те атрибуты, по которым вы действительно часто выполняете запросы. Каждый материализованный столбец требует дополнительного места в хранилище и ресурсов при вставке, поэтому здесь работает принцип «материализуйте то, чем пользуетесь», а не «материализуйте всё подряд».

  ```sql
  ALTER TABLE otel_logs
    ADD COLUMN ServiceVersion LowCardinality(String)
    MATERIALIZED LogAttributes['service.version'];
  ```

  Для существующих строк новый столбец останется пустым, пока вы дополнительно не выполните `ALTER TABLE otel_logs MATERIALIZE COLUMN ServiceVersion`.

  Подробнее: [Материализация часто используемых атрибутов](/use-cases/observability/clickstack/performance_tuning#materialize-frequently-queried-attributes).

  ## Добавьте индекс пропуска данных \{#add-skip-indexes\}

  Индекс пропуска данных позволяет ClickHouse исключать гранулы данных, которые заведомо не подходят под фильтр, превращая полное сканирование в небольшое адресное чтение. Стоит знать о трёх типах:

  * **Текстовые индексы** (`text(tokenizer = ...)`) для строковых столбцов и массивов `mapKeys`/`*AttributeItems`. В стандартную схему журналов они уже входят.
  * **Min-max-индексы** для числовых столбцов, по которым фильтрация идёт по диапазону. Классический пример — `Duration` в трейсах.
  * **Bloom-фильтры** для проверок на равенство при высокой кардинальности в версиях ClickHouse, которые ещё не поддерживают текстовые индексы.

  ```sql
  ALTER TABLE otel_traces ADD INDEX idx_duration Duration TYPE minmax GRANULARITY 1;
  ALTER TABLE otel_traces MATERIALIZE INDEX idx_duration;
  ```

  Индекс пропуска данных окупает стоимость своей проверки только тогда, когда он действительно отсекает гранулы. Подтвердите это с помощью `EXPLAIN indexes = 1` на репрезентативном запросе, прежде чем считать, что он помог.

  Подробнее: [Добавление индекса пропуска данных](/use-cases/observability/clickstack/performance_tuning#adding-skip-indexes).

  ## Предварительно вычисляйте повторяющиеся агрегации \{#materialized-views\}

  Когда одна и та же агрегация снова и снова используется на панели мониторинга (топ сервисов по доле ошибок, p99 задержки по конечной точке, число запросов в минуту), materialized view вычисляет результат во время вставки и записывает его в небольшую rollup-таблицу. После этого панели мониторинга обращаются к rollup, а не к сырым журналам или трейсам, что обходится значительно дешевле.

  Это оправданно, когда панель мониторинга активно используется, а исходная таблица велика. Цена — дополнительные затраты CPU во время вставки и необходимость поддерживать вторую таблицу.

  Подробнее: [Использование materialized view](/use-cases/observability/clickstack/performance_tuning#exploiting-materialized-views).

  ## Выберите первичный ключ под свой шаблон доступа \{#choose-primary-key\}

  Первичный ключ определяет, как строки сортируются на диске. Фильтры по ведущим столбцам этого ключа позволяют ClickHouse сразу переходить к нужному диапазону данных; фильтры, которые не начинаются с одного из этих столбцов, приводят к сканированию всей партиции.

  Ключ журналов по умолчанию `(toStartOfFiveMinutes(Timestamp), ServiceName, Timestamp)` оптимизирован под сценарий «что произошло за последние N минут для сервиса X». Если большинство ваших запросов начинается с другого столбца (идентификатор арендатора, идентификатор клиента, регион), изменение первичного ключа так, чтобы он начинался с этого столбца, — самое эффективное изменение, которое вы можете внести.

  ```sql
  CREATE TABLE otel_logs_v2
  (
    -- те же столбцы, что и у otel_logs
  )
  ENGINE = MergeTree
  ORDER BY (TenantId, ServiceName, Timestamp);
  ```

  ClickHouse не позволяет редактировать первичный ключ на месте, поэтому здесь нужна миграция таблицы, а не простой `ALTER`. В руководстве по настройке производительности пошагово показано, как создать новую таблицу, перенаправить ингестию и использовать таблицу `Merge`, чтобы существующие панели мониторинга продолжали работать и со старыми, и с новыми данными.

  Подробнее: [Изменение первичного ключа](/use-cases/observability/clickstack/performance_tuning#modifying-the-primary-key).
