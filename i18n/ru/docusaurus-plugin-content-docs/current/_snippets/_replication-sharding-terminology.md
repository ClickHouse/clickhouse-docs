## Терминология {#terminology}

### Реплика {#replica}

Копия данных. В ClickHouse всегда существует как минимум одна копия ваших данных, поэтому минимальное количество **реплик** равно одной. Это важная особенность: вы можете быть не привыкли считать исходную копию данных репликой, но именно этот термин используется в коде и документации ClickHouse. Добавление второй реплики данных обеспечивает отказоустойчивость.

### Шард {#shard}

Подмножество данных. В ClickHouse всегда существует как минимум один шард для ваших данных, поэтому если вы не разделяете данные между несколькими серверами, ваши данные будут храниться в одном шарде. Шардирование данных между несколькими серверами может использоваться для распределения нагрузки при превышении мощности одного сервера. Целевой сервер определяется **ключом шардирования**, который задается при создании распределенной таблицы. Ключ шардирования может быть случайным или представлять собой результат [хеш-функции](/sql-reference/functions/hash-functions). В примерах развертывания с шардированием в качестве ключа шардирования будет использоваться `rand()`, а также будет предоставлена дополнительная информация о том, когда и как выбрать другой ключ шардирования.

### Распределенная координация {#distributed-coordination}

ClickHouse Keeper предоставляет систему координации для репликации данных и выполнения распределенных DDL-запросов. ClickHouse Keeper совместим с Apache ZooKeeper.
