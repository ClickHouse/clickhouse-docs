## Проблема \{#problem\}

ClickHouse Cloud не поддерживает транзакции из нескольких операторов в традиционном для реляционных СУБД смысле.
Из-за этого возникают две распространенные проблемы:

1. Атомарность массовой загрузки в рамках одной таблицы: распространенный подход заключается во вставке во временные частичные ключи, после чего записи копируются в основной ключ, а временные записи удаляются. Этот подход работает неэффективно — особенно на этапе удаления, который может занимать более 90% общего времени выполнения операции.
2. Согласованность между несколькими таблицами: когда конвейер успешно загружает Table A, но завершается с ошибкой на Table B, изменения в Table A уже зафиксированы и не могут быть откатаны. Аналитики, выполняющие запросы к обеим таблицам, видят несогласованные данные.

## Общие сведения \{#background\}

ClickHouse гарантирует атомарность на уровне одной операции вставки в одну партицию: если `INSERT` завершается успешно, видимыми становятся все строки этого блока; если он завершается с ошибкой, не видна ни одна из них. Однако встроенного механизма, позволяющего атомарно зафиксировать данные сразу в нескольких операциях вставки или в нескольких таблицах, нет.

Команды управления партициями ([`MOVE PARTITION TO TABLE`](/sql-reference/statements/alter/partition#move-partition-to-table), [`REPLACE PARTITION`](/sql-reference/statements/alter/partition#replace-partition), [`ATTACH PARTITION FROM`](/sql-reference/statements/alter/partition#attach-partition-from)) работают на уровне метаданных, если исходная и целевая таблицы используют одну и ту же политику хранения.

Это означает, что они выполняются практически мгновенно независимо от объема данных, поэтому идеально подходят как базовые элементы для паттернов атомарной замены.

## Рекомендуемое решение \{#recommended-solution\}

Вместо того чтобы выполнять вставку напрямую в продукционные таблицы и пытаться устранять последствия в случае сбоя, используйте выделенные staging-таблицы как промежуточную зону. После проверки данных используйте операции на уровне партиций, чтобы атомарно перенести данные в продукционные таблицы.

## Пошаговое руководство по атомарности одной таблицы \{#step-by-step\}

<VerticalStepper headerLevel="h3">
  ### Создайте staging-таблицу {#create-staging-table}

  Используйте ту же схему, ключ партиционирования, `ORDER BY` и политику хранения, что и у продукционной таблицы.

  ```sql
  CREATE TABLE my_table_staging AS my_table_prod;
  ```

  ### Вставьте данные в staging-таблицу {#insert-into-staging}

  Выполняйте вставку в staging-таблицу, а не в продукционную.

  ```sql
  INSERT INTO my_table_staging SELECT ... FROM source;
  ```

  ### Если вставка завершается ошибкой, выполните TRUNCATE и повторите попытку {#on-failure}

  Очистите staging-таблицу и запустите загрузку снова. Данные в продукционной таблице при этом не затрагиваются.

  ```sql
  TRUNCATE TABLE my_table_staging;
  ```

  ### Если вставка выполнена успешно, переместите партиции в продукционную таблицу {#on-success}

  Чтобы переместить данные в продукционную таблицу и удалить их из staging-таблицы, используйте `MOVE PARTITION`.

  ```sql
  ALTER TABLE my_table_staging MOVE PARTITION <partition_expr> TO TABLE my_table_prod;
  ```

  Либо скопируйте данные в существующую партицию продукционной таблицы с помощью `ATTACH PARTITION`.

  ```sql
  ALTER TABLE my_table_prod ATTACH PARTITION tuple() FROM my_table_staging;
  ```

  Обе операции выполняются на уровне метаданных в рамках одной и той же политики хранения и завершаются почти мгновенно.

  ### Очистите staging-таблицу {#clean-up}

  После перемещения всех партиций очистите staging-таблицу, чтобы она оставалась пустой для следующей загрузки.

  ```sql
  TRUNCATE TABLE my_table_staging;
  ```
</VerticalStepper>

## Согласованность нескольких таблиц {#multi-table-consistency}

Тот же подход работает и для конвейеров, в которых две или более таблицы должны быть полностью загружены, прежде чем хоть одна из них станет доступна аналитикам. Загрузите данные каждой таблицы в её собственную staging-таблицу, проверьте их все, а затем выполните перемещение партиций одновременно. Поскольку каждое такое перемещение — это почти мгновенная операция с метаданными, окно, в котором таблицы оказываются несогласованными, сокращается с длительности полной загрузки до времени, необходимого для переключения партиций.

## Требования и ограничения {#requirements-and-constraints}

Для `MOVE PARTITION TO TABLE`, `REPLACE PARTITION` и `ATTACH PARTITION FROM` исходная и целевая таблицы должны иметь:

* Одинаковую структуру столбцов
* Одинаковый ключ партиционирования, ключ `ORDER BY` и первичный ключ
* Одинаковую политику хранения
* Целевая таблица должна содержать все индексы и проекции исходной таблицы

## Пример

Вы можете попробовать это на [fiddle](https://fiddle.clickhouse.com/7ef9ed84-ac14-4f2c-9ca5-d5913089769a):

```sql
CREATE TABLE prod
(
  uid Int16,
  name String,
  age Int16
)
ENGINE=MergeTree
ORDER BY ();

CREATE TABLE staging
(
  uid Int16,
  name String,
  age Int16
)
ENGINE=MergeTree
ORDER BY ();

-- Initial data
INSERT INTO prod VALUES (123, 'John', 33);
INSERT INTO prod VALUES (456, 'Ksenia', 48);
-- Load data
INSERT INTO staging VALUES (8811, 'Alice', 50);
INSERT INTO staging VALUES (8812, 'Bob', 23);

-- Validate import
SELECT 'Staging count:', COUNT() FROM staging;
-- Move partition
ALTER TABLE staging MOVE PARTITION tuple() TO TABLE prod; -- atomic op

-- Check data
SELECT 'Prod count:', COUNT() FROM prod;
SELECT * FROM prod;
```
