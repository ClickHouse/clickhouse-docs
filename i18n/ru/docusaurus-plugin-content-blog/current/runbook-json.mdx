:::note
Тип столбца JSON готов к использованию в продакшене, начиная с ClickHouse 25.3+. Более ранние версии не рекомендуются для использования в продакшене.
:::

Ваши данные поступают в формате JSON. ClickHouse предоставляет несколько способов их хранения — от полностью типизированных столбцов до сырого String. Правильный выбор зависит от того, насколько предсказуема ваша схема и нужны ли вам запросы по отдельным полям.

**Область охвата:** На этой странице рассматриваются решения по проектированию схемы для хранения данных JSON. Она не охватывает [JSON-форматы ввода/вывода](/interfaces/formats/JSON), [JSON-функции](/sql-reference/functions/json-functions) или синтаксис запросов. Подробнее о самом типе столбца JSON см. в разделе [Используйте JSON там, где это уместно](/best-practices/use-json-where-appropriate).

**Предполагается:** знакомство с [созданием таблиц в ClickHouse](/sql-reference/statements/create/table), основами [MergeTree](/engines/table-engines/mergetree-family/mergetree) и синтаксисом типов столбцов.

## Быстрый выбор \{#quick-decision\}

* **Если** у каждого поля есть известный, стабильный тип и схема меняется редко
  **→** [Типизированные столбцы](#typed-columns)
* **Если** большинство полей стабильны, но какая-то часть данных динамическая или непредсказуемая
  **→** [Гибрид (типизированные столбцы + JSON)](#hybrid)
* **Если** вся структура динамическая и ключи появляются и исчезают от записи к записи
  **→** [Нативный JSON-столбец](#native-json)
* **Если** динамические поля — это пары ключ-значение с единообразным типом значений (например, строковые теги, числовые метрики)
  **→** [`Map`](#when-map-fits-better) вместо JSON
* **Если** вы только храните и извлекаете JSON-блоб без запросов к отдельным полям
  **→** [Непрозрачное хранение в String](#opaque-storage)

:::note
Не путайте JSON *формат* и JSON *тип столбца*. Вы можете выполнять вставку данных в формате JSON (через `JSONEachRow` и т. п.) в типизированные столбцы, вообще не используя тип столбца `JSON`. Здесь речь идет о типах столбцов, а не о входных форматах.
:::

## Детали подхода \{#approach-details\}

### Типизированные столбцы \{#typed-columns\}

**Когда использовать:** Структура JSON полностью известна на этапе проектирования. Поля и типы не меняются от записи к записи. Даже сложные вложенные структуры (массивы объектов, вложенные map-структуры) можно выразить с помощью типов [`Array`](/sql-reference/data-types/array), [`Tuple`](/sql-reference/data-types/tuple) и [`Nested`](/sql-reference/data-types/nested-data-structures/nested).

**Компромиссы:** Для изменения схемы требуется `ALTER TABLE`. Неожиданные поля при вставке молча отбрасываются, если схема не была обновлена.

<Details summary="Настройка, проверка и подводные камни">
  **Настройка**

  ```sql
  CREATE TABLE events
  (
      `timestamp` DateTime,
      `service`   LowCardinality(String),
      `level`     Enum8('DEBUG' = 1, 'INFO' = 2, 'WARN' = 3, 'ERROR' = 4),
      `message`   String,
      `host`      LowCardinality(String),
      `duration_ms` UInt32
  )
  ENGINE = MergeTree
  ORDER BY (service, timestamp)
  ```

  **Проверка**

  ```sql
  -- Убедитесь, что типы столбцов соответствуют ожидаемым
  DESCRIBE TABLE events FORMAT Vertical

  -- Выполните вставку и запрос, чтобы проверить, что схема корректно обрабатывает ваши данные
  INSERT INTO events FORMAT JSONEachRow
  {"timestamp":"2025-03-19 10:00:00","service":"api","level":"INFO","message":"request handled","host":"node-1","duration_ms":42}

  SELECT service, level, duration_ms FROM events WHERE service = 'api'
  ```

  **Обратите внимание**

  * Если вы вставляете данные JSON с `JSONEachRow` и JSON содержит поля, которых нет в схеме, ClickHouse по умолчанию молча их отбрасывает. Установите [`input_format_skip_unknown_fields`](/operations/settings/formats#input_format_skip_unknown_fields) в `0`, если хотите вместо этого получать ошибки.
</Details>

### Гибридный подход (типизированные столбцы + JSON) \{#hybrid\}

**Когда использовать:** Основной набор полей стабилен (таймстэмпы, ID, коды статуса), но часть данных динамична. Например, это пользовательские атрибуты, теги, метаданные или поля расширения, которые различаются от записи к записи.

**Компромиссы:** Максимальная производительность на типизированных столбцах и гибкость в JSON-столбце. При этом JSON-столбец по-прежнему добавляет накладные расходы при вставке и увеличивает затраты на хранение динамической части данных.

<Details summary="Настройка, проверка и подводные камни">
  **Настройка**

  ```sql
  CREATE TABLE events
  (
      `timestamp`  DateTime,
      `service`    LowCardinality(String),
      `level`      Enum8('DEBUG' = 1, 'INFO' = 2, 'WARN' = 3, 'ERROR' = 4),
      `message`    String,
      `host`       LowCardinality(String),
      `duration_ms` UInt32,
      `attributes` JSON(
          max_dynamic_paths = 256,
          `http.status_code` UInt16,
          `http.method` LowCardinality(String),
          SKIP REGEXP 'debug\..*'
      )
  )
  ENGINE = MergeTree
  ORDER BY (service, timestamp)
  ```

  **Проверка**

  ```sql
  -- Вставьте выборку данных и проверьте обнаруженные пути
  INSERT INTO events FORMAT JSONEachRow
  {"timestamp":"2025-03-19 10:00:00","service":"api","level":"INFO","message":"request handled","host":"node-1","duration_ms":42,"attributes":{"http.status_code":200,"http.method":"GET","user.region":"eu-west","custom_tag":"abc"}}

  SELECT JSONAllPathsWithTypes(attributes)
  FROM events
  FORMAT PrettyJSONEachRow
  ```

  **Обратите внимание**

  * Используйте [подсказки типов](/sql-reference/data-types/newjson) для JSON-путей, которые известны заранее. Подсказки обходят столбец-дискриминатор и сохраняют путь как обычный типизированный столбец — с той же производительностью и без дополнительных накладных расходов.
  * Используйте `SKIP` или `SKIP REGEXP` для путей, которые вы никогда не запрашиваете (отладочные метаданные, внутренние ID трассировки), чтобы экономить место в хранилище и уменьшать число подстолбцов.
  * Устанавливайте `max_dynamic_paths` пропорционально количеству различных путей, которые вы действительно запрашиваете. Значение по умолчанию (1024) подходит для большинства случаев. Уменьшите его, если динамическая часть у вас невелика.
  * Не устанавливайте `max_dynamic_paths` выше 10 000. Большие значения увеличивают потребление ресурсов и снижают эффективность.

  :::note Ключи с точками
  Ключи с точками (например, `http.status_code`) по умолчанию обрабатываются как вложенные пути, поэтому `{"http.status_code": 200}` сохраняется так же, как `{"http": {"status_code": 200}}`. Это часто встречается в атрибутах OTel. Используйте подсказки типов, чтобы управлять тем, как сохраняются пути с точками, или включите `json_type_escape_dots_in_keys` (25.8+).
  :::
</Details>

### Нативный JSON-столбец \{#native-json\}

**Когда использовать:** Если структура действительно непредсказуема и ключи появляются и исчезают от записи к записи. Пользовательские схемы, системы плагинов или ингестия в data lake, где вы не контролируете исходную схему.

**Компромиссы:** Вставка медленнее, чем в типизированные столбцы. Чтение объекта целиком медленнее, чем из String. Есть дополнительные накладные расходы на хранение из-за управления подстолбцами. Хорошо подходит для запросов к отдельным полям по конкретным путям.

<Details summary="Настройка, проверка и подводные камни">
  **Настройка**

  ```sql
  CREATE TABLE dynamic_events
  (
      `id`   UInt64,
      `ts`   DateTime DEFAULT now(),
      `data` JSON(
          max_dynamic_paths = 512,
          `event_type` LowCardinality(String),
          `version` UInt8
      )
  )
  ENGINE = MergeTree
  ORDER BY (data.event_type, ts)
  ```

  Используйте формат [`JSONAsObject`](/interfaces/formats/JSONAsObject) при вставке JSON-документов целиком в JSON-столбец. Он обрабатывает каждую входную строку как полноценный JSON-объект и сопоставляет её со столбцом.

  **Проверка**

  ```sql
  INSERT INTO dynamic_events (id, data) FORMAT JSONEachRow
  {"id": 1, "data": {"event_type": "click", "version": 2, "page": "/home", "button_id": "cta-1"}}
  {"id": 2, "data": {"event_type": "purchase", "version": 1, "item_id": "SKU-99", "amount": 49.99, "currency": "USD"}}

  -- Проверьте, какие пути обнаружил ClickHouse и какие типы им присвоил
  SELECT JSONAllPathsWithTypes(data) FROM dynamic_events FORMAT PrettyJSONEachRow

  -- Запрос по конкретному пути
  SELECT data.page FROM dynamic_events WHERE data.event_type = 'click'
  ```

  **На что обратить внимание**

  * Без подсказок типов ClickHouse определяет типы для каждого пути по первым увиденным значениям. Если `score` приходит как `"10"` (строка) в одной записи и как `10` (целое число) в другой, для этого пути создаётся столбец-дискриминатор, и запросы становятся медленнее. Добавляйте подсказки для путей с известными типами.
  * Когда число путей превышает `max_dynamic_paths`, избыточные значения перемещаются в [общую структуру данных](/sql-reference/data-types/newjson#shared-data-structure), что снижает производительность запросов. Контролируйте это с помощью [`JSONDynamicPaths()`](/sql-reference/data-types/newjson#introspection-functions) и держите лимит ниже 10 000.
  * Каждый динамический путь поддерживает до `max_dynamic_types` (по умолчанию 32) различных типов данных. Если один путь превышает этот предел, дополнительные типы переводятся в общее Variant-хранилище. Обычно это несущественно, если только в ваших данных нет сильно расходящихся типов для одного и того же поля.
</Details>

### Хранение в виде непрозрачной строки \{#opaque-storage\}

**Когда использовать:** JSON-документы сохраняются и извлекаются целиком, а затем передаются в приложение, архивируются или отправляются дальше по конвейеру. Без фильтрации или агрегации по отдельным полям внутри ClickHouse.

**Компромиссы:** Максимально быстрые вставки и самая простая схема. Запросы по отдельным полям невозможны без разбора во время выполнения (семейство `JSONExtract`), а это медленно при больших объёмах.

<Details summary="Настройка, проверка и подводные камни">
  **Настройка**

  ```sql
  CREATE TABLE raw_events
  (
      `id`        UInt64,
      `received`  DateTime DEFAULT now(),
      `payload`   String
  )
  ENGINE = MergeTree
  ORDER BY (received)
  ```

  **Проверка**

  ```sql
  INSERT INTO raw_events (id, payload) VALUES
  (1, '{"type":"click","page":"/home"}'),
  (2, '{"type":"purchase","item":"SKU-99","amount":49.99}')

  -- Убедитесь, что данные возвращаются без изменений
  SELECT payload FROM raw_events WHERE id = 1

  -- Убедитесь, что при необходимости поля всё ещё можно разбирать произвольно
  SELECT JSONExtractString(payload, 'type') AS event_type FROM raw_events
  ```

  **Обратите внимание**

  * Если требования изменятся и позже вам понадобятся запросы по отдельным полям, потребуется создать новую таблицу с типизированными столбцами или JSON-столбцами и выполнить дозаполнение данных. Если есть хотя бы вероятность, что вы будете запрашивать отдельные поля, лучше сразу выбрать [гибридный подход](#hybrid).
  * Функции `JSONExtract` разбирают строку при каждом запросе. Это приемлемо для разового анализа, но не для дашбордов в продакшен или рабочих нагрузок с высоким QPS.
  * Если JSON-документы в столбце `payload` имеют большой размер, рассмотрите использование кодеков сжатия (`ZSTD`) для столбца String — такие данные хорошо сжимаются.
</Details>

## Сравнение \{#comparison\}

| Характеристика                 | Типизированные столбцы      | Гибрид                                                 | Нативный JSON                                     | String                                 |
| ------------------------------ | --------------------------- | ------------------------------------------------------ | ------------------------------------------------- | -------------------------------------- |
| **Производительность вставки** | Самая высокая               | Высокая                                                | Умеренная                                         | Самая высокая                          |
| **Запросы по отдельным полям** | Самые быстрые               | Быстрые (типизированные); хорошие (JSON с подсказками) | Хорошие (с подсказками); медленнее (динамические) | Медленные (разбор во время выполнения) |
| **Чтение объекта целиком**     | Быстро                      | Умеренно                                               | Медленно                                          | Быстрее всего                          |
| **Эффективность хранения**     | Лучшая                      | Хорошая                                                | Умеренная                                         | Хорошая (хорошо сжимается)             |
| **Гибкость схемы**             | Отсутствует (`ALTER TABLE`) | Частичная (жесткое ядро, гибкая остальная часть)       | Полная                                            | Полная                                 |
| **Сложность**                  | Низкая                      | Средняя                                                | Средняя–высокая                                   | Низкая                                 |

## Когда Map подходит лучше \{#when-map-fits-better\}

Если ваши динамические поля представляют собой однородные пары «ключ-значение», то есть все значения имеют один и тот же тип, [`Map(String, T)`](/sql-reference/data-types/map) будет проще и эффективнее, чем JSON-столбец. Типичные примеры: строковые теги (`Map(String, String)`), числовые метрики (`Map(String, Float64)`) или флаги возможностей (`Map(String, Bool)`).

```sql
CREATE TABLE tagged_events
(
    `timestamp` DateTime,
    `service`   LowCardinality(String),
    `tags`      Map(String, String)  -- e.g., {"env": "prod", "region": "us-east-1", "team": "platform"}
)
ENGINE = MergeTree
ORDER BY (service, timestamp)
```

`Map` поддерживает фильтрацию на уровне ключей (`tags['env'] = 'prod'`), требует меньше места для хранения, чем JSON, и позволяет избежать накладных расходов на подстолбцы типа JSON. Обратите внимание: по умолчанию поиск по ключам выполняется линейным сканированием `Map` — это нормально для небольших наборов тегов, но для `Map` со 100+ ключами стоит рассмотреть [`with_buckets`-сериализацию](/sql-reference/data-types/map#bucketed-map-serialization). Используйте JSON, если значения имеют смешанные типы или структура содержит вложенность, — используйте `Map`, если это плоские пары ключ-значение с единым типом значений.
