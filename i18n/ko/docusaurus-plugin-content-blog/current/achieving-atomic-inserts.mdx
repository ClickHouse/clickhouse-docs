## 문제 \{#problem\}

ClickHouse Cloud는 전통적인 RDBMS의 의미에서 여러 SQL 문으로 이루어진 트랜잭션을 지원하지 않습니다.
이로 인해 흔히 다음과 같은 두 가지 문제가 발생합니다.

1. 대량 적재 시 단일 테이블 원자성: 일반적으로는 임시 부분 키에 먼저 삽입한 뒤, 레코드를 실제 키로 복사하고 임시 레코드를 삭제합니다. 하지만 이 방식은 성능이 좋지 않으며, 특히 삭제 단계가 전체 작업 시간의 90% 이상을 차지할 수 있습니다.
2. 다중 테이블 일관성: 파이프라인이 테이블 A에는 성공적으로 적재했지만 테이블 B에서 실패하면, 테이블 A는 이미 커밋되어 롤백할 수 없습니다. 두 테이블을 함께 쿼리하는 분석가는 서로 맞지 않는 데이터를 보게 됩니다.

## 배경 \{#background\}

ClickHouse는 단일 삽입, 단일 파티션 단위에서 원자성을 보장합니다. 즉, `INSERT`가 성공하면 해당 블록의 모든 행을 볼 수 있고, 실패하면 어떤 행도 볼 수 없습니다. 그러나 여러 번의 삽입이나 여러 테이블에 걸친 데이터를 원자적으로 커밋하는 기본 제공 메커니즘은 없습니다.

파티션 조작 명령([`MOVE PARTITION TO TABLE`](/sql-reference/statements/alter/partition#move-partition-to-table), [`REPLACE PARTITION`](/sql-reference/statements/alter/partition#replace-partition), [`ATTACH PARTITION FROM`](/sql-reference/statements/alter/partition#attach-partition-from))은 원본 테이블과 대상 테이블이 동일한 스토리지 정책을 공유할 때 메타데이터 수준에서 수행됩니다.

즉, 데이터 크기와 관계없이 거의 즉시 실행되므로 원자적 스왑 패턴을 구현하는 데 이상적인 구성 요소입니다.

## 권장 방법 \{#recommended-solution\}

실패 시 정리 작업을 해야 하는 프로덕션 테이블에 직접 데이터를 삽입하는 대신, 전용 스테이징 테이블을 중간 적재 영역으로 사용합니다. 데이터를 검증한 후에는 파티션 수준 작업을 사용해 해당 데이터를 운영 테이블로 원자적으로 승격합니다.

## 단일 테이블 원자성을 위한 단계별 가이드 \{#step-by-step\}

<VerticalStepper headerLevel="h3">
  ### 스테이징 테이블 생성 {#create-staging-table}

  프로덕션 테이블과 동일한 스키마(schema), 파티션 키, `ORDER BY`, 스토리지 정책을 사용합니다.

  ```sql
  CREATE TABLE my_table_staging AS my_table_prod;
  ```

  ### 스테이징 테이블에 데이터 삽입 {#insert-into-staging}

  프로덕션 테이블 대신 스테이징 테이블에 데이터를 삽입합니다.

  ```sql
  INSERT INTO my_table_staging SELECT ... FROM source;
  ```

  ### 삽입이 실패하면 TRUNCATE하고 다시 시도 {#on-failure}

  스테이징 테이블을 TRUNCATE한 후 다시 적재를 실행합니다. 프로덕션 데이터에는 영향을 주지 않습니다.

  ```sql
  TRUNCATE TABLE my_table_staging;
  ```

  ### 삽입이 성공하면 파티션을 프로덕션으로 이동 {#on-success}

  데이터를 프로덕션으로 이동하고 스테이징 테이블에서 제거하려면 `MOVE PARTITION`을 사용합니다.

  ```sql
  ALTER TABLE my_table_staging MOVE PARTITION <partition_expr> TO TABLE my_table_prod;
  ```

  또는 `ATTACH PARTITION`을 사용해 데이터를 프로덕션의 기존 파티션으로 복사할 수 있습니다.

  ```sql
  ALTER TABLE my_table_prod ATTACH PARTITION tuple() FROM my_table_staging;
  ```

  두 작업 모두 동일한 스토리지 정책 내에서 수행되는 메타데이터 수준 변경이므로 거의 즉시 완료됩니다.

  ### 스테이징 테이블 정리 {#clean-up}

  모든 파티션을 이동한 후에는 다음 적재를 위해 스테이징 테이블을 비워 두도록 TRUNCATE합니다.

  ```sql
  TRUNCATE TABLE my_table_staging;
  ```
</VerticalStepper>

## 다중 테이블 일관성 {#multi-table-consistency}

동일한 패턴을 사용하면 2개 이상의 테이블이 모두 적재되기 전에는 어떤 테이블도 분석가에게 노출되지 않아야 하는 파이프라인도 처리할 수 있습니다. 각 테이블의 데이터를 전용 스테이징 테이블에 적재하고 모두 검증한 다음, 파티션 이동 작업을 함께 실행하십시오. 각 이동 작업은 거의 즉시 완료되는 메타데이터 작업이므로, 테이블 간 불일치가 발생할 수 있는 구간은 전체 적재 시간 수준에서 파티션을 교체하는 데 걸리는 시간 수준으로 줄어듭니다.

## 요구 사항 및 제약 조건 {#requirements-and-constraints}

`MOVE PARTITION TO TABLE`, `REPLACE PARTITION`, `ATTACH PARTITION FROM`의 경우, 원본 테이블과 대상 테이블은 다음 요건을 충족해야 합니다:

* 컬럼 구조가 동일해야 합니다
* 파티션 키, `ORDER BY` 키, 프라이머리 키가 동일해야 합니다
* 스토리지 정책이 동일해야 합니다
* 대상 테이블에는 원본 테이블의 모든 인덱스와 프로젝션이 포함되어야 합니다

## 예시

아래 예시는 [fiddle](https://fiddle.clickhouse.com/7ef9ed84-ac14-4f2c-9ca5-d5913089769a)에서 대화형으로 실행할 수 있습니다:

```sql
CREATE TABLE prod
(
  uid Int16,
  name String,
  age Int16
)
ENGINE=MergeTree
ORDER BY ();

CREATE TABLE staging
(
  uid Int16,
  name String,
  age Int16
)
ENGINE=MergeTree
ORDER BY ();

-- Initial data
INSERT INTO prod VALUES (123, 'John', 33);
INSERT INTO prod VALUES (456, 'Ksenia', 48);
-- Load data
INSERT INTO staging VALUES (8811, 'Alice', 50);
INSERT INTO staging VALUES (8812, 'Bob', 23);

-- Validate import
SELECT 'Staging count:', COUNT() FROM staging;
-- Move partition
ALTER TABLE staging MOVE PARTITION tuple() TO TABLE prod; -- atomic op

-- Check data
SELECT 'Prod count:', COUNT() FROM prod;
SELECT * FROM prod;
```
