:::note
JSON 컬럼 타입은 ClickHouse 25.3+부터 프로덕션 환경에서 사용할 수 있습니다. 이전 버전은 프로덕션 용도로 권장되지 않습니다.
:::

데이터는 JSON으로 유입됩니다. ClickHouse는 완전히 타입이 지정된 컬럼부터 원시 `String`까지, 이를 저장할 수 있는 여러 가지 방법을 제공합니다. 적절한 선택은 schema를 얼마나 예측 가능하게 유지할 수 있는지, 그리고 필드 단위 쿼리가 필요한지에 따라 달라집니다.

**범위:** 이 페이지에서는 JSON 데이터를 저장하기 위한 schema 설계 결정을 다룹니다. [JSON 입력/출력 형식](/interfaces/formats/JSON), [JSON 함수](/sql-reference/functions/json-functions), 또는 쿼리 구문은 다루지 않습니다. JSON 컬럼 타입 자체에 대한 배경 정보는 [적절한 경우 JSON 사용](/best-practices/use-json-where-appropriate)을 참조하십시오.

**가정:** [ClickHouse 테이블 생성](/sql-reference/statements/create/table), [MergeTree](/engines/table-engines/mergetree-family/mergetree)의 기본 개념, 그리고 컬럼 타입 구문에 익숙하다고 가정합니다.

## 빠른 선택 \{#quick-decision\}

* **모든 필드의 타입이 알려져 있고 안정적이며 schema가 거의 변경되지 않는 경우**
  **→** [타입이 지정된 컬럼](#typed-columns)
* **대부분의 필드는 안정적이지만 일부 섹션은 동적이거나 예측하기 어려운 경우**
  **→** [하이브리드(타입 지정 + JSON)](#hybrid)
* **전체 구조가 동적이며, 레코드마다 키가 나타났다 사라지는 경우**
  **→** [네이티브 JSON 컬럼](#native-json)
* **동적 필드가 일관된 값 타입(예: 문자열 태그, 숫자 메트릭)을 갖는 key-value 쌍인 경우**
  **→** JSON 대신 [`Map`](#when-map-fits-better)
* **필드 수준 쿼리 없이 JSON blob만 저장하고 조회하는 경우**
  **→** [불투명 String 저장](#opaque-storage)

:::note
JSON *형식*과 JSON *컬럼 타입*을 혼동하지 마십시오. `JSON` 컬럼 타입을 전혀 사용하지 않고도 JSON 형식의 데이터를(`JSONEachRow` 등을 통해) 타입이 지정된 컬럼에 삽입할 수 있습니다. 여기서의 선택은 입력 형식이 아니라 컬럼 타입에 관한 것입니다.
:::

## 접근 방식 상세 \{#approach-details\}

### 타입이 지정된 컬럼 \{#typed-columns\}

**이용하는 타이밍:** JSON 구조를 설계 시점에 완전히 알고 있는 경우입니다. 필드와 타입이 레코드마다 바뀌지 않습니다. 배열 안의 객체나 중첩된 맵 같은 복잡한 중첩 구조도 [`Array`](/sql-reference/data-types/array), [`Tuple`](/sql-reference/data-types/tuple), [`Nested`](/sql-reference/data-types/nested-data-structures/nested) 타입으로 표현할 수 있습니다.

**트레이드오프:** schema를 변경하려면 `ALTER TABLE`이 필요합니다. schema를 업데이트하지 않으면 예상하지 못한 필드는 삽입 시 별도 경고 없이 제거됩니다.

<Details summary="설정, 검증 및 주의사항">
  **설정**

  ```sql
  CREATE TABLE events
  (
      `timestamp` DateTime,
      `service`   LowCardinality(String),
      `level`     Enum8('DEBUG' = 1, 'INFO' = 2, 'WARN' = 3, 'ERROR' = 4),
      `message`   String,
      `host`      LowCardinality(String),
      `duration_ms` UInt32
  )
  ENGINE = MergeTree
  ORDER BY (service, timestamp)
  ```

  **검증**

  ```sql
  -- 컬럼 타입이 예상과 일치하는지 확인합니다
  DESCRIBE TABLE events FORMAT Vertical

  -- 삽입과 쿼리를 통해 schema가 데이터를 올바르게 처리하는지 검증합니다
  INSERT INTO events FORMAT JSONEachRow
  {"timestamp":"2025-03-19 10:00:00","service":"api","level":"INFO","message":"request handled","host":"node-1","duration_ms":42}

  SELECT service, level, duration_ms FROM events WHERE service = 'api'
  ```

  **주의할 점**

  * `JSONEachRow`로 JSON 데이터를 삽입할 때 JSON에 schema에 없는 필드가 포함되어 있으면, ClickHouse는 기본적으로 해당 필드를 자동으로 무시합니다. 대신 오류가 발생하게 하려면 [`input_format_skip_unknown_fields`](/operations/settings/formats#input_format_skip_unknown_fields)를 `0`으로 설정하십시오.
</Details>

### 하이브리드(타입이 지정된 컬럼 + JSON) \{#hybrid\}

**이용하는 타이밍:** 핵심 필드 집합은 안정적이지만(타임스탬프, ID, 상태 코드) 페이로드의 일부는 동적일 때 사용합니다. 예를 들어 사용자 정의 속성, 태그, 메타데이터, 또는 레코드마다 달라지는 확장 필드가 여기에 해당합니다.

**트레이드오프:** 타입이 지정된 컬럼에서는 최대 성능을 얻고, JSON 컬럼에서는 유연성을 확보할 수 있습니다. JSON 컬럼의 동적 부분에는 여전히 삽입 오버헤드와 저장 비용이 발생합니다.

<Details summary="설정, 검증, 주의사항">
  **설정**

  ```sql
  CREATE TABLE events
  (
      `timestamp`  DateTime,
      `service`    LowCardinality(String),
      `level`      Enum8('DEBUG' = 1, 'INFO' = 2, 'WARN' = 3, 'ERROR' = 4),
      `message`    String,
      `host`       LowCardinality(String),
      `duration_ms` UInt32,
      `attributes` JSON(
          max_dynamic_paths = 256,
          `http.status_code` UInt16,
          `http.method` LowCardinality(String),
          SKIP REGEXP 'debug\..*'
      )
  )
  ENGINE = MergeTree
  ORDER BY (service, timestamp)
  ```

  **검증**

  ```sql
  -- 샘플 데이터를 삽입하고 추론된 경로를 확인합니다
  INSERT INTO events FORMAT JSONEachRow
  {"timestamp":"2025-03-19 10:00:00","service":"api","level":"INFO","message":"request handled","host":"node-1","duration_ms":42,"attributes":{"http.status_code":200,"http.method":"GET","user.region":"eu-west","custom_tag":"abc"}}

  SELECT JSONAllPathsWithTypes(attributes)
  FROM events
  FORMAT PrettyJSONEachRow
  ```

  **주의할 점**

  * 미리 알고 있는 JSON 경로에는 [타입 힌트](/sql-reference/data-types/newjson)를 사용하십시오. 힌트를 사용하면 discriminator 컬럼을 우회하고 해당 경로를 일반적인 타입이 지정된 컬럼처럼 저장하므로, 동일한 성능을 얻으면서도 오버헤드가 없습니다.
  * 쿼리하지 않는 경로(디버그 메타데이터, 내부 추적 ID)에는 `SKIP` 또는 `SKIP REGEXP`를 사용하여 저장 공간을 절약하고 서브컬럼 수를 줄이십시오.
  * `max_dynamic_paths`는 실제로 쿼리하는 고유 경로 수에 맞춰 설정하십시오. 기본값(1024)은 대부분의 경우에 적합합니다. 동적 영역이 작다면 더 낮게 설정하십시오.
  * `max_dynamic_paths`를 10,000보다 크게 설정하지 마십시오. 값이 너무 크면 리소스 소비가 증가하고 효율이 떨어집니다.

  :::note 점이 포함된 키
  점이 포함된 키(예: `http.status_code`)는 기본적으로 중첩 경로로 처리되므로, `{"http.status_code": 200}`는 `{"http": {"status_code": 200}}`와 동일하게 저장됩니다. 이는 OTel 속성에서 흔히 나타납니다. 타입 힌트를 사용해 점이 포함된 경로의 저장 방식을 제어하거나, `json_type_escape_dots_in_keys` (25.8+)를 활성화하십시오.
  :::
</Details>

### 네이티브 JSON 컬럼 \{#native-json\}

**이용하는 타이밍:** 구조를 실제로 예측할 수 없고, 레코드마다 키가 생겼다 사라지는 경우에 적합합니다. 사용자 정의 schema, 플러그인 시스템, 또는 업스트림 schema를 제어할 수 없는 데이터 레이크 수집 환경이 여기에 해당합니다.

**장단점:** 타입이 지정된 컬럼보다 삽입이 느립니다. 전체 객체 읽기 성능은 String보다 느립니다. 서브컬럼 관리에 따른 스토리지 오버헤드가 있습니다. 특정 경로에 대한 필드 수준 쿼리에는 효과적입니다.

<Details summary="설정, 확인 및 주의 사항">
  **설정**

  ```sql
  CREATE TABLE dynamic_events
  (
      `id`   UInt64,
      `ts`   DateTime DEFAULT now(),
      `data` JSON(
          max_dynamic_paths = 512,
          `event_type` LowCardinality(String),
          `version` UInt8
      )
  )
  ENGINE = MergeTree
  ORDER BY (data.event_type, ts)
  ```

  JSON 컬럼에 전체 JSON 문서를 삽입할 때는 [`JSONAsObject`](/interfaces/formats/JSONAsObject) 형식을 사용하십시오. 이 형식은 각 입력 줄을 해당 컬럼에 매핑되는 완전한 JSON 객체로 처리합니다.

  **확인**

  ```sql
  INSERT INTO dynamic_events (id, data) FORMAT JSONEachRow
  {"id": 1, "data": {"event_type": "click", "version": 2, "page": "/home", "button_id": "cta-1"}}
  {"id": 2, "data": {"event_type": "purchase", "version": 1, "item_id": "SKU-99", "amount": 49.99, "currency": "USD"}}

  -- ClickHouse가 감지한 경로와 해당 타입 확인
  SELECT JSONAllPathsWithTypes(data) FROM dynamic_events FORMAT PrettyJSONEachRow

  -- 특정 경로 쿼리
  SELECT data.page FROM dynamic_events WHERE data.event_type = 'click'
  ```

  **주의할 사항**

  * 타입 힌트가 없으면 ClickHouse는 각 경로에서 처음 들어온 값을 기준으로 타입을 추론합니다. 한 레코드에서 `score`가 `"10"`(문자열)으로 들어오고 다른 레코드에서는 `10`(정수)으로 들어오면, 해당 경로에 discriminator 컬럼이 생성되어 쿼리가 더 느려집니다. 타입이 정해진 경로에는 힌트를 추가하십시오.
  * 경로 수가 `max_dynamic_paths`를 초과하면 초과된 값은 쿼리 성능이 낮아지는 [공유 데이터 구조](/sql-reference/data-types/newjson#shared-data-structure)로 이동합니다. [`JSONDynamicPaths()`](/sql-reference/data-types/newjson#introspection-functions)로 모니터링하고, 한도는 10,000 미만으로 유지하십시오.
  * 각 동적 경로는 최대 `max_dynamic_types`(기본값 32)개의 서로 다른 데이터 타입을 지원합니다. 단일 경로가 이를 초과하면 추가 타입은 shared variant storage로 대체됩니다. 같은 필드에서 데이터 타입이 매우 일관되지 않은 경우가 아니라면, 일반적으로는 큰 문제가 되지 않습니다.
</Details>

### 불투명 String 저장 방식 \{#opaque-storage\}

**이용하는 타이밍:** JSON 문서를 통째로 저장하고 그대로 조회한 다음, 애플리케이션으로 전달하거나 보관하거나 다운스트림으로 전달하는 경우에 적합합니다. ClickHouse 내부에서는 필드 수준 필터링이나 집계를 수행하지 않습니다.

**장단점:** 가장 빠르게 삽입할 수 있고 schema도 가장 단순합니다. 런타임 파싱(`JSONExtract` 계열) 없이는 필드 수준 쿼리를 수행할 수 없으며, 규모가 커질수록 성능이 느려집니다.

<Details summary="설정, 검증 및 주의사항">
  **설정**

  ```sql
  CREATE TABLE raw_events
  (
      `id`        UInt64,
      `received`  DateTime DEFAULT now(),
      `payload`   String
  )
  ENGINE = MergeTree
  ORDER BY (received)
  ```

  **검증**

  ```sql
  INSERT INTO raw_events (id, payload) VALUES
  (1, '{"type":"click","page":"/home"}'),
  (2, '{"type":"purchase","item":"SKU-99","amount":49.99}')

  -- 데이터가 손상 없이 그대로 조회되는지 확인합니다
  SELECT payload FROM raw_events WHERE id = 1

  -- 필요할 때 필드를 즉석에서 파싱할 수 있는지 확인합니다
  SELECT JSONExtractString(payload, 'type') AS event_type FROM raw_events
  ```

  **주의할 점**

  * 요구 사항이 바뀌어 나중에 필드 수준 쿼리가 필요해지면, 타입이 지정된 컬럼 또는 JSON 컬럼이 있는 새 테이블을 만들고 데이터를 backfill해야 합니다. 개별 필드를 쿼리할 가능성이 조금이라도 있다면, 대신 [하이브리드 접근 방식](#hybrid)으로 시작하십시오.
  * `JSONExtract` 함수는 쿼리할 때마다 문자열을 파싱합니다. 즉석 탐색에는 괜찮지만, 프로덕션 대시보드나 높은 QPS의 워크로드에는 적합하지 않습니다.
  * JSON 페이로드가 크다면 String 컬럼에 압축 코덱(`ZSTD`) 적용을 고려하십시오 — 압축 효율이 좋습니다.
</Details>

## 비교 \{#comparison\}

| 항목             | 타입이 지정된 컬럼         | 하이브리드                    | 네이티브 JSON          | String     |
| -------------- | ------------------ | ------------------------ | ------------------ | ---------- |
| **삽입 처리량**     | 가장 빠름              | 빠름                       | 보통                 | 가장 빠름      |
| **필드 단위 쿼리**   | 가장 빠름              | 빠름(타입 지정), 양호함(힌트된 JSON) | 양호함(힌트 지정), 느림(동적) | 느림(런타임 파싱) |
| **전체 객체 읽기**   | 빠름                 | 보통                       | 느림                 | 가장 빠름      |
| **저장 효율성**     | 최상                 | 좋음                       | 보통                 | 좋음(압축이 잘됨) |
| **schema 유연성** | 없음 (`ALTER TABLE`) | 부분적(핵심은 엄격하고, 나머지는 유연함)  | 완전                 | 완전         |
| **복잡도**        | 낮음                 | 중간                       | 중간–높음              | 낮음         |

## Map이 더 적합한 경우 \{#when-map-fits-better\}

동적 필드가 모두 동일한 타입의 key-value 쌍으로 이루어져 있다면 [`Map(String, T)`](/sql-reference/data-types/map)은 JSON 컬럼보다 더 단순하고 효율적입니다. 일반적인 예시로는 문자열 태그(`Map(String, String)`), 숫자 메트릭(`Map(String, Float64)`), 또는 기능 플래그(`Map(String, Bool)`)가 있습니다.

```sql
CREATE TABLE tagged_events
(
    `timestamp` DateTime,
    `service`   LowCardinality(String),
    `tags`      Map(String, String)  -- e.g., {"env": "prod", "region": "us-east-1", "team": "platform"}
)
ENGINE = MergeTree
ORDER BY (service, timestamp)
```

`Map`은 키 수준 필터링(`tags['env'] = 'prod'`)을 지원하고, JSON보다 저장 비용이 적게 들며, JSON 타입의 서브컬럼 오버헤드를 피할 수 있습니다. 기본적으로 키 조회는 맵을 선형 탐색하므로, 작은 태그 집합에는 문제가 없지만 키가 100개 이상인 맵에서는 [`with_buckets` 직렬화](/sql-reference/data-types/map#bucketed-map-serialization)를 고려하십시오. 값의 타입이 혼합되어 있거나 구조가 중첩된 경우에는 JSON을 사용하고, 구조가 평평하며 값 타입이 균일한 키-값 쌍인 경우에는 `Map`을 사용하십시오.
