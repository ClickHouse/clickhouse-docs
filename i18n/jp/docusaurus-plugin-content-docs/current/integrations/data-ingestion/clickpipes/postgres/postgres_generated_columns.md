レプリケーション対象となっているテーブルで PostgreSQL の生成列を使用する場合、いくつか重要な点に留意する必要があります。こうした落とし穴は、レプリケーション処理やレプリケーション先システムにおけるデータ整合性に影響を与える可能性があります。



## 生成列の問題 {#the-problem-with-generated-columns}

1. **`pgoutput`経由で公開されない:** 生成列は`pgoutput`論理レプリケーションプラグインを通じて公開されません。つまり、PostgreSQLから別のシステムへデータをレプリケートする際、生成列の値はレプリケーションストリームに含まれません。

2. **主キーに関する問題:** 生成列が主キーの一部である場合、レプリケーション先で重複排除の問題が発生する可能性があります。生成列の値はレプリケートされないため、レプリケーション先システムは行を適切に識別し重複排除するために必要な情報を持ちません。

3. **スキーマ変更に関する問題:** すでにレプリケートされているテーブルに生成列を追加した場合、新しい列はレプリケーション先に反映されません。これはPostgreSQLが新しい列のRelationMessageを提供しないためです。その後、同じテーブルに新しい非生成列を追加すると、ClickPipeがスキーマの調整を試みる際に、レプリケーション先で生成列を見つけることができず、レプリケーションプロセスが失敗します。


## ベストプラクティス {#best-practices}

これらの制限を回避するには、以下のベストプラクティスを考慮してください：

1. **レプリケーション先で生成カラムを再作成する：** レプリケーションプロセスに生成カラムの処理を任せるのではなく、dbt（data build tool）などのツールやその他のデータ変換メカニズムを使用して、レプリケーション先でこれらのカラムを再作成することを推奨します。

2. **プライマリキーに生成カラムを使用しない：** レプリケーション対象となるテーブルを設計する際は、プライマリキーの一部として生成カラムを含めないようにしてください。


## UIの今後の改善 {#upcoming-improvements-to-ui}

今後のバージョンでは、以下の機能を持つUIを追加する予定です:

1. **生成カラムを含むテーブルの識別:** 生成カラムを含むテーブルを識別する機能を提供します。これにより、ユーザーはどのテーブルがこの問題の影響を受けるかを把握できます。

2. **ドキュメントとベストプラクティス:** レプリケートされたテーブルで生成カラムを使用する際のベストプラクティスを提供し、一般的な問題を回避する方法に関するガイダンスを含めます。
