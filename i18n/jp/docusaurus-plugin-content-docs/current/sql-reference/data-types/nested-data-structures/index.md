ネストされたデータ構造は、セルの中にあるテーブルのようなものです。ネストされたデータ構造のパラメータ (カラム名や型) は、[CREATE TABLE](../../../sql-reference/statements/create/table.md) クエリの場合と同様に指定します。各テーブル行は、ネストされたデータ構造内の任意の数の行に対応できます。

:::tip[カラム名でのドット使用を避ける]
ドットを含むカラム名、ドットで始まる共通プレフィックスを持つカラム、および `Array` 型のカラムは、`flatten_nested = 1` (デフォルト) の場合に、それぞれフラット化された Nested 構造の一部として解釈される可能性があります。これにより、挿入時の配列長検証やリネームに関する制約が予期せず発生することがあります。

可能であれば、カラム名にドットを使用するのは避けてください。
意図的に `Nested` のセマンティクスが必要な場合を除き、カラム名にはドットの代わりにアンダースコア (`_`) や他のセパレーターを使用してください。
:::

例:

```sql
CREATE TABLE test.visits(
  CounterID UInt32,
  StartDate Date,
  Sign Int8,
  IsNew UInt8,
  VisitID UInt64,
  UserID UInt64,
--highlight-start
  Goals Nested(
    ID UInt32,
    Serial UInt32,
    EventTime DateTime,
    Price Int64,
    OrderID String,
    CurrencyID UInt32
  )
--highlight-end
)
ENGINE = CollapsingMergeTree(Sign)
ORDER BY (StartDate, intHash32(UserID), (CounterID, StartDate, intHash32(UserID), VisitID));
```

上記の `CREATE TABLE` DDLステートメントでは、コンバージョン、つまり達成された目標に関するデータを含む、ネストされたデータ構造 `Goals` を宣言しています。
&#39;visits&#39; テーブルの各行は、0 件以上のコンバージョンに対応します。

[`flatten_nested`](/operations/settings/settings#flatten_nested) 設定を `0` にすると (デフォルトは `flatten_nested=1`) 、任意のレベルのネストがサポートされます。

通常、ネストされたデータ構造を扱う場合、そのカラムはドット区切りのカラム名で指定します。
これらのカラムは、対応する型の配列を構成します。
1 つのネストされたデータ構造に属するすべてのカラム配列は、同じ長さになります。

たとえば:

```sql
SELECT
    Goals.ID,
    Goals.EventTime
FROM test.visits
WHERE CounterID = 101500 AND length(Goals.ID) < 5
ORDER BY VisitID
LIMIT 10
```

### WHERE 句での Nested カラムのフィルタリング \{#filtering-nested-columns-in-where\}

`Nested` 構造の各カラムは `Array` として格納されるため、`WHERE` 句で参照すると、個々の要素ではなく各行に対して配列全体が返されます。ネストされたカラムをスカラー値と直接比較することはできないため、代わりに [array functions](/sql-reference/functions/array-functions) を使う必要があります。

たとえば、次のクエリは単に行を返さないのではなく、例外をスローします。これは `Goals.ID` の型が `Array(UInt32)` であり、`equals(Array(UInt32), UInt32)` が有効な比較ではないためです。

```sql
-- WRONG: compares the entire Array to a scalar
SELECT * FROM test.visits
WHERE Goals.ID = 591325;
```

```text
Code: 43. DB::Exception: Illegal types of arguments (`Array(UInt32)`, `UInt32`)
of function `equals`. (ILLEGAL_TYPE_OF_ARGUMENT)
```

配列に特定の値が含まれているかどうかを確認するには、[`has`](/sql-reference/functions/array-functions#has) を使用します：

```sql
-- Find visits that have at least one goal with ID 591325
SELECT CounterID, VisitID, Goals.ID
FROM test.visits
WHERE has(Goals.ID, 591325);
```

条件がより複雑な場合は、[`arrayExists`](/sql-reference/functions/array-functions#arrayExists) を使用します。

```sql
-- Find visits that have at least one goal with ID greater than 1000000
SELECT CounterID, VisitID, Goals.ID
FROM test.visits
WHERE arrayExists(id -> id > 1000000, Goals.ID);
```

`length` で配列の長さを基準に絞り込んだり、`notEmpty` で空の配列を除外したりできます。

```sql
-- Visits with at least 3 goals
SELECT CounterID, VisitID, Goals.ID
FROM test.visits
WHERE length(Goals.ID) >= 3;

-- Visits with at least one goal (non-empty array)
SELECT CounterID, VisitID, Goals.ID
FROM test.visits
WHERE notEmpty(Goals.ID);
```

ネストされた構造の個々の要素に対してフィルタし、行全体を対象にしないようにするには、まず `ARRAY JOIN` を使って配列を展開します。
`ARRAY JOIN` の後は各要素がそれぞれ別の行になるため、`WHERE` 句はスカラー値に適用されます。
詳しくは、[`ARRAY JOIN` 句](/sql-reference/statements/select/array-join) を参照してください。例:

```sql
SELECT
    Goal.ID,
    Goal.EventTime
FROM test.visits
ARRAY JOIN Goals AS Goal
WHERE CounterID = 101500 AND length(Goals.ID) < 5
ORDER BY VisitID, Goal.Serial
LIMIT 10
```

### データの挿入 \{#inserting-data\}

`INSERT` クエリでは、ネストされたデータ構造を構成する各カラムの配列を、それぞれ個別のカラム配列であるかのように別々に渡す必要があります。挿入時には、それらの長さがすべて同じであることがシステムによって確認されます。

ネストされた各サブカラムは、ドット記法 (`Goals.ID`, `Goals.Serial`, ...) を使ってカラム一覧に記載し、対応する値は配列で指定します:

```sql
INSERT INTO test.visits
    (CounterID, StartDate, Sign, IsNew, VisitID, UserID,
     Goals.ID, Goals.Serial, Goals.EventTime, Goals.Price, Goals.OrderID, Goals.CurrencyID)
VALUES
    -- A visit with two goals: each nested sub-column gets an array of length 2
    (101500, '2014-03-18', 1, 1, 2001, 200001,
     [1073752, 591325], [1, 2],
     ['2014-03-18 10:00:00', '2014-03-18 10:05:00'],
     [100, 200], ['order_a', 'order_b'], [1, 2]),
    -- A visit with no goals: all nested sub-columns get empty arrays
    (101500, '2014-03-18', 1, 0, 2002, 200002,
     [], [], [], [], [], []);
```

1つの行内にあるネストされたすべてのサブカラム配列は、同じ長さである必要があります。長さが一致しない場合はエラーになります：

```sql
-- ERROR: Goals.ID has 2 elements, but Goals.Serial has 1
INSERT INTO test.visits
    (CounterID, StartDate, Sign, IsNew, VisitID, UserID,
     Goals.ID, Goals.Serial, Goals.EventTime, Goals.Price, Goals.OrderID, Goals.CurrencyID)
VALUES
    (101500, '2014-03-18', 1, 1, 2003, 200003,
     [1073752, 591325], [1],
     ['2014-03-18 12:00:00'], [0], [''], [0]);
```

`DESCRIBE` クエリでは、ネストされたデータ構造内のカラムも同様に個別に表示されます。

### ALTER の制限 \{#alter-limitations\}

ネストされたデータ構造に対する `ALTER` クエリには、次の制限があります。

**サブカラムの追加** は通常どおり可能です。既存の `Nested` 構造に新しいサブカラムを追加できます。

```sql
ALTER TABLE test.visits ADD COLUMN Goals.Revenue Float64;
```

**サブカラムの削除**は、個々のサブカラムに対して実行できます：

```sql
ALTER TABLE test.visits DROP COLUMN Goals.Revenue;
```

**サブカラムの型の変更**は可能で、mutation (データの書き換え) がトリガーされます：

```sql
ALTER TABLE test.visits MODIFY COLUMN Goals.Price Int32;
```

**名前の変更** には制限があります。同じネスト構造内であれば、サブカラムの名前を変更できます。

```sql
-- OK: stays within the Goals structure
ALTER TABLE test.visits RENAME COLUMN Goals.Price TO Goals.Amount;
```

ただし、以下の操作は**できません**:

* ネスト構造全体の名前を変更すること (例: `Goals` を `Conversions` に変更) 。
* サブカラムを別のネスト構造に移動すること (例: `Goals.ID` を `OtherNested.ID` に移動) 。
* サブカラムをネスト構造の外に出したり、ネスト構造内に移動したりすること (例: `Goals.ID` を `GoalID` に変更、またはその逆) 。
