## 説明 \{#description\}

`system.metrics` および `system.events` テーブルのメトリクス値の履歴を保持し、一定間隔でディスクにフラッシュされます。

## カラム \{#columns\}

{/*AUTOGENERATED_START*/ }

* `hostname` ([LowCardinality(String)](../../sql-reference/data-types/string.md)) — クエリを実行するサーバーのホスト名。
* `event_date` ([Date](../../sql-reference/data-types/date.md)) — イベントの日付。
* `event_time` ([DateTime](../../sql-reference/data-types/datetime.md)) — イベントの時刻。
* `event_time_microseconds` ([DateTime64](../../sql-reference/data-types/datetime64.md)) — マイクロ秒精度のイベント時刻。

{/*AUTOGENERATED_END*/ }

**Schema**
このテーブルは、XML タグ `<schema_type>` を使用して、異なる schema タイプに設定できます。デフォルトの schema タイプは `wide` で、各メトリクスまたはプロファイルイベントはそれぞれ отдель個別のカラムに格納されます。この schema は、単一カラムの読み取りにおいて最も高性能かつ効率的です。

`transposed` schema では、`system.asynchronous_metric_log` と同様のフォーマットでデータを格納し、メトリクスとイベントは行として保存されます。この schema は、マージ時のリソース消費を抑えられるため、リソースが限られた構成に適しています。
