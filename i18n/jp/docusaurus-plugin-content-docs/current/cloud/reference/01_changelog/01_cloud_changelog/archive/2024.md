## 2024年12月20日 \{#december-20-2024\}

### マーケットプレイスサブスクリプションの組織への紐付け \{#marketplace-subscription-organization-attachment\}

既存の ClickHouse Cloud 組織に、新しいマーケットプレイスサブスクリプションを紐付けられるようになりました。マーケットプレイスでのサブスクリプション登録を完了し、ClickHouse Cloud にリダイレクトされたら、以前に作成した既存の組織を新しいマーケットプレイスサブスクリプションに紐付けることができます。以降、その組織内のリソースはマーケットプレイス経由で課金されます。

<Image img={add_marketplace} size="md" alt="既存の組織にマーケットプレイスサブスクリプションを追加する方法を示す ClickHouse Cloud インターフェイス" border />

### OpenAPI キーの有効期限を必須にする \{#force-openapi-key-expiration\}

API キーの有効期限の選択肢を制限し、有効期限のない OpenAPI キーを作成できないようにできるようになりました。組織でこの制限を有効にするには、ClickHouse Cloud Support チームにお問い合わせください。

### 通知用のカスタムメールアドレス \{#custom-emails-for-notifications\}

組織管理者は、特定の通知に対して追加の受信者としてメールアドレスを追加できるようになりました。これは、通知をエイリアス宛てに送信したい場合や、ClickHouse Cloudのユーザーではない組織内のほかのユーザーにも送信したい場合に便利です。設定するには、Cloud Consoleの通知設定に移動し、メール通知の送信先にしたいメールアドレスを編集します。

## 2024年12月6日 \{#december-6-2024\}

### BYOC (ベータ) \{#byoc-beta\}

AWS 向けの Bring Your Own Cloud は現在ベータ提供中です。このデプロイモデルでは、お客様ご自身の AWS アカウントで ClickHouse Cloud をデプロイして実行できます。現在、11 以上の AWS リージョンでのデプロイをサポートしており、対応リージョンは今後さらに追加される予定です。利用をご希望の場合は、[サポートにお問い合わせください](https://clickhouse.com/support/program)。なお、このデプロイは大規模な導入向けに限定されています。

### ClickPipes の Postgres CDC (変更データキャプチャ) コネクタ \{#postgres-change-data-capture-cdc-connector-in-clickpipes\}

このすぐに利用できる統合機能により、数回クリックするだけで Postgres データベースを ClickHouse Cloud にレプリケートし、ClickHouse を活用した高速分析を実現できます。このコネクタは、Postgres からの継続的なレプリケーションにも、一度限りの移行にも使用できます。

### ダッシュボード (ベータ) \{#dashboards-beta\}

今週、ClickHouse Cloud でダッシュボードのベータ版をリリースしたことをお知らせします。ダッシュボードを使用すると、保存済みクエリから可視化を作成し、それらをダッシュボードに整理して配置し、クエリパラメータを使用してダッシュボードを操作できます。利用を開始するには、[ダッシュボードのドキュメント](/cloud/manage/dashboards)を参照してください。

<Image img={beta_dashboards} size="lg" alt="新しいダッシュボードのベータ機能と可視化を表示する ClickHouse Cloud インターフェイス" border />

### クエリ API エンドポイント (GA) \{#query-api-endpoints-ga\}

ClickHouse Cloud でクエリ API エンドポイントが GA リリースされたことをお知らせします。クエリ API エンドポイントを使用すると、保存済みクエリ用の RESTful API エンドポイントを数クリックで立ち上げられ、言語クライアントの扱いや認証の複雑さに煩わされることなく、アプリケーションでデータの利用を開始できます。初回リリース以降、次のような改善を行ってきました。

* API エンドポイントのレイテンシを低減 (特にコールドスタート時)
* API エンドポイントの RBAC 制御を強化
* CORS で許可するドメインを設定可能
* 結果のストリーミング
* ClickHouse 互換のすべての出力フォーマットをサポート

これらの改善に加え、既存のフレームワークを活用して、ClickHouse Cloud のサービスに対して任意の SQL クエリを実行できる汎用クエリ API エンドポイントも発表します。汎用 API エンドポイントは、サービスの設定ページから有効化および設定できます。

使い始めるには、[クエリ API エンドポイントのドキュメント](/cloud/get-started/query-endpoints)を参照してください。

<Image img={api_endpoints} size="lg" alt="さまざまな設定が表示された API エンドポイント設定画面を示す ClickHouse Cloud インターフェイス" border />

### ネイティブJSONサポート (ベータ) \{#native-json-support-beta\}

ClickHouse Cloud でネイティブJSONサポートのベータ提供を開始します。利用を開始するには、support[クラウドサービスで有効にするには](/cloud/support)までお問い合わせください。

### ベクトル類似度索引を使用したベクトル検索 (早期アクセス) \{#vector-search-using-vector-similarity-indexes-early-access\}

近似ベクトル検索向けのベクトル類似度索引を、早期アクセスとして提供開始します。

ClickHouse はすでに、幅広い[距離関数]https://clickhouse.com/blog/reinvent-2024-product-announcements#vector-search-using-vector-similarity-indexes-early-access)と線形スキャンを実行する機能により、ベクトルベースのユースケースを強力にサポートしています。さらに最近では、[usearch](https://github.com/unum-cloud/usearch) ライブラリと Hierarchical Navigable Small Worlds (HNSW) 近似最近傍探索アルゴリズムを利用した、実験的な[近似ベクトル検索](/engines/table-engines/mergetree-family/annindexes)手法を追加しました。

使い始めるには、[早期アクセスのウェイトリストにご登録ください](https://clickhouse.com/cloud/vector-search-index-waitlist)。

## 2024年10月4日 \{#october-4-2024\}

### ClickHouse Cloud が GCP 向けに HIPAA 対応サービスのベータ提供を開始 \{#clickhouse-cloud-now-offers-hipaa-ready-services-in-beta-for-gcp\}

保護対象保健情報 (PHI) に対してより高いセキュリティを求めるお客様は、[Google Cloud Platform (GCP)](https://cloud.google.com/) 上の ClickHouse Cloud を利用開始できるようになりました。ClickHouse は、[HIPAA Security Rule](https://www.hhs.gov/hipaa/for-professionals/security/index.html) で定められている管理上、物理上、および技術上の保護措置を講じており、現在では、お客様固有のユースケースやワークロードに応じて適用できる、構成可能なセキュリティ設定も提供しています。利用可能なセキュリティ設定の詳細については、[Security features page](/cloud/security) をご確認ください。

本サービスは、GCP `us-central-1` の **Dedicated** サービスタイプをご利用のお客様を対象に提供されており、Business Associate Agreement (BAA) が必要です。この機能へのアクセスをリクエストする場合、または今後追加される GCP、AWS、Azure の各リージョン向けウェイトリストへの参加を希望する場合は、[sales](mailto:sales@clickhouse.com) または [support](https://clickhouse.com/support/program) までお問い合わせください。

### Compute-compute separation が GCP と Azure でもプライベートプレビューで利用可能になりました \{#compute-compute-separation-is-now-in-private-preview-for-gcp-and-azure\}

先日、AWS 向けの Compute-Compute Separation のプライベートプレビューを発表しましたが、このたび GCP と Azure でも利用可能になりました。

Compute-compute separation を使用すると、特定のサービスを読み書き用または読み取り専用のサービスとして指定できるため、コストとパフォーマンスを最適化するアプリケーション向けに、最適なコンピュート設定を設計できます。詳細については、[ドキュメント](/cloud/reference/warehouses)を参照してください。

### セルフサービスによるMFAリカバリーコード \{#self-service-mfa-recovery-codes\}

多要素認証を使用しているお客様は、スマートフォンを紛失した場合やトークンを誤って削除した場合に備えて使用できるリカバリーコードを取得できるようになりました。初めてMFAを登録するお客様には、セットアップ時にこのコードが提供されます。すでにMFAを使用しているお客様は、現在のMFAトークンを削除して新しいトークンを追加することで、リカバリーコードを取得できます。

### ClickPipes のアップデート: カスタム証明書、レイテンシの可視化、その他 \{#clickpipes-update-custom-certificates-latency-insights-and-more\}

ClickPipes は、ClickHouse サービスにデータをインジェストする最も簡単な方法です。今回は、その ClickPipes の最新アップデートをご紹介します。これらの新機能は、データインジェストの制御性を高めるとともに、パフォーマンスメトリクスの可視性を向上させるよう設計されています。

*Kafka 向けカスタム認証証明書*

Kafka 向け ClickPipes で、SASL とパブリック SSL/TLS を使用する Kafka ブローカー向けのカスタム認証証明書がサポートされるようになりました。ClickPipe の設定時に SSL Certificate セクションから独自の証明書を簡単にアップロードでき、Kafka への接続をより安全に確立できます。

*Kafka と Kinesis のレイテンシメトリクスを導入*

パフォーマンスの可視性は重要です。ClickPipes にレイテンシグラフが追加され、メッセージが生成されてから (Kafka Topic または Kinesis Stream) 、ClickHouse Cloud にインジェストされるまでの時間を把握できるようになりました。この新しいメトリクスにより、データパイプラインのパフォーマンスをより詳細に監視し、それに応じて最適化できます。

<Image img={latency_insights} size="lg" alt="データインジェストのパフォーマンスに関するレイテンシメトリクスのグラフを表示する ClickPipes インターフェイス" border />

<br />

*Kafka と Kinesis のスケーリング制御 (プライベートベータ)&#x20;*

高い処理量に対応するには、データ量やレイテンシ要件を満たすために追加のリソースが必要になる場合があります。ClickPipes では、クラウドコンソールから直接利用できる水平スケーリングを導入します。この機能は現在プライベートベータとして提供されており、要件に応じてより効果的にリソースをスケールできます。ベータへの参加をご希望の場合は、[サポート](https://clickhouse.com/support/program) にお問い合わせください。

*Kafka と Kinesis の生メッセージインジェスト*

Kafka または Kinesis のメッセージ全体を、解析せずに取り込めるようになりました。ClickPipes では、`_raw_message` [仮想カラム](/integrations/clickpipes/kafka/reference/#kafka-virtual-columns) がサポートされ、メッセージ全体を 1 つの String カラムにマッピングできます。これにより、必要に応じて生データを柔軟に扱えます。

## 2024年8月15日 \{#august-15-2024\}

### Compute-compute separation が AWS でプライベートプレビューになりました \{#compute-compute-separation-is-now-in-private-preview-for-aws\}

既存の ClickHouse Cloud サービスでは、レプリカが読み取りと書き込みの両方を処理しており、特定のレプリカを一方の操作のみ処理するように設定することはできません。近日提供予定の新機能 Compute-compute separation では、特定のサービスを読み書き可能または読み取り専用のサービスとして指定できるため、コストとパフォーマンスを最適化するアプリケーション向けの最適なコンピュート構成を設計できます。

この新しい Compute-compute separation 機能では、それぞれが独自のエンドポイントを持つ複数のコンピュートノードグループを作成でき、同じオブジェクトストレージフォルダーを使用することで、同じテーブルやビューなどを共有できます。詳しくは、[Compute-compute separation についてはこちら](/cloud/reference/warehouses) を参照してください。この機能のプライベートプレビューへのアクセスを希望する場合は、[サポートにお問い合わせください](https://clickhouse.com/support/program)。

### S3 および GCS 向け ClickPipes が GA となり、Continuous mode をサポート \{#clickpipes-for-s3-and-gcs-now-in-ga-continuous-mode-support\}

### Kafka ClickPipes Connector の設定可能なオフセット (ベータ) \{#configurable-offsets-for-kafka-clickpipes-connector-beta\}

最近まで、新しい [ClickPipes 向け Kafka Connector](/integrations/clickpipes/kafka) をセットアップすると、常に Kafka トピックの先頭からデータを消費していました。そのため、履歴データを再処理したい場合や、新しく到着するデータのみを監視したい場合、あるいは特定の位置から再開したい場合などには、十分な柔軟性がないことがありました。

使用できるオプションは次のとおりです。

* 先頭から: Kafka トピックの先頭からデータの消費を開始します。すべての履歴データを再処理する必要がある場合に適しています。
* 最新から: 最新のオフセットからデータの消費を開始します。新しいメッセージのみに関心がある場合に便利です。
* タイムスタンプから: 特定のタイムスタンプ時点以降に生成されたメッセージからデータの消費を開始します。この機能により、よりきめ細かく制御でき、正確な時点から処理を再開できます。

### 新しい ClickHouse Cloud コンソールの紹介 \{#introducing-the-new-clickhouse-cloud-console\}

このリリースでは、新しい Cloud Console のプライベートプレビューを提供します。

ClickHouse では、開発者体験をどう向上させるかを常に追求しています。最速のリアルタイムデータウェアハウスを提供するだけでは不十分であり、使いやすく管理しやすいことも重要だと考えています。

### 信頼性に関する変更 \{#reliability-changes\}

* ユーザーに影響する後方互換性のない変更: 以前は、特定の条件下で 2 つの機能 ([is&#95;deleted](/engines/table-engines/mergetree-family/replacingmergetree#is_deleted) と `OPTIMIZE CLEANUP`) により、ClickHouse 内のデータが破損する可能性がありました。機能の中核は維持しつつ、ユーザーデータの完全性を保護するため、この機能の動作を調整しました。具体的には、MergeTree の設定 `clean_deleted_rows` は現在非推奨であり、もはや効果はありません。`CLEANUP` キーワードはデフォルトでは使用できません (使用するには `allow_experimental_replacing_merge_with_cleanup` を有効にする必要があります) 。`CLEANUP` を使用する場合は、必ず `FINAL` と組み合わせて使用してください。また、`OPTIMIZE FINAL CLEANUP` を実行した後に、古いバージョンの行が挿入されないことを保証する必要があります。
