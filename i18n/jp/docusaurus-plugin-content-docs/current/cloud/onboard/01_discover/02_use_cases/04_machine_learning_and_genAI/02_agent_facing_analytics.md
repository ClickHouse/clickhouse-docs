## エージェント向け分析の概念 {#agent-facing-analytics}

### 「エージェント」とは {#agents}

AIエージェントは、単純なタスク実行(関数呼び出し)を超えて進化したデジタルアシスタントと考えることができます。文脈を理解し、意思決定を行い、特定の目標に向けて意味のある行動を取ることができます。エージェントは「感知-思考-行動」のループで動作し(ReActエージェントを参照)、様々な入力(テキスト、メディア、データ)を処理し、状況を分析した上で、その情報を活用して有用な処理を実行します。最も重要な点として、アプリケーション領域に応じて、理論的には様々なレベルの自律性で動作でき、人間の監督が必要な場合もあれば不要な場合もあります。

ここでの転換点となったのは、大規模言語モデル(LLM)の登場です。AIエージェントという概念自体はかなり以前から存在していましたが、GPTシリーズのようなLLMにより、「理解」とコミュニケーションの能力が飛躍的に向上しました。まるで突然「人間らしさ」に流暢になったかのように、リクエストを把握し、モデルの学習から得られた関連する文脈情報を用いて応答できるようになったのです。

### AIエージェントの強力な機能:「ツール」 {#tools}

これらのエージェントは「ツール」へのアクセスによって真価を発揮します。ツールはAIエージェントにタスクを実行する能力を与えることで機能を強化します。単なる会話インターフェースではなく、数値計算、情報検索、顧客コミュニケーションの管理など、実際に作業を完遂できるようになりました。問題の解決方法を説明できる人と、実際に問題を解決できる人との違いと考えてください。

例えば、ChatGPTは現在、デフォルトで検索ツールが搭載されています。検索プロバイダーとの統合により、モデルは会話中にウェブから最新情報を取得できます。これにより、応答の事実確認、最近の出来事やデータへのアクセスが可能になり、学習データのみに依存せず、最新の情報を提供できるようになります。

<Image img={ml_ai_05} alt='Agents equipped with tools' size='md' />

ツールは、検索拡張生成(RAG)パイプラインの実装を簡素化するためにも使用できます。AIモデルが学習時に習得した知識のみに依存するのではなく、RAGはモデルが応答を生成する前に関連情報を取得できるようにします。例を挙げます:カスタマーサポートを支援するAIアシスタントの使用(例:Salesforce AgentForce、ServiceNow AI Agents)。RAGがない場合、一般的な学習内容のみを使用して質問に答えます。しかしRAGを使用すると、顧客が最新の製品機能について質問した際、システムは応答を作成する前に最新のドキュメント、リリースノート、過去のサポートチケットを取得します。これにより、回答はAIモデルが利用可能な最新情報に基づいたものになります。

### 推論モデル {#reasoning-models}

AI分野におけるもう一つの発展、そしておそらく最も興味深いものの一つが、推論モデルの登場です。OpenAI o1、Anthropic Claude、DeepSeek-R1のようなシステムは、プロンプトに応答する前に「思考」ステップを導入することで、より体系的なアプローチを取ります。即座に回答を生成するのではなく、推論モデルはChain-of-Thought(CoT)のようなプロンプト技術を使用して、問題を複数の角度から分析し、ステップに分解し、必要に応じて利用可能なツールを使用して文脈情報を収集します。

これは、推論と実用的なツールの組み合わせによって、より複雑なタスクを処理できる高度なシステムへの移行を表しています。この分野における最新の例の一つが、OpenAIのdeep researchの導入です。これは、オンラインで複雑な多段階の調査タスクを自律的に実行できるエージェントです。テキスト、画像、PDFを含む様々なソースから情報を処理・統合し、5分から30分以内に包括的なレポートを生成します。これは従来、人間が数時間かけて行っていたタスクです。

## AIエージェント向けリアルタイム分析 {#real-time-analytics-for-ai-agents}

企業のCRMデータを含むリアルタイム分析データベースにアクセス可能なエージェント型AIアシスタントのケースを考えてみましょう。ユーザーが最新(分単位)の売上トレンドについて質問すると、AIアシスタントは接続されたデータソースにクエリを実行します。そして、前月比成長率、季節変動、新興製品カテゴリなどの意味のあるパターンやトレンドを特定するために、データを反復的に分析します。最後に、主要な発見を説明する自然言語の応答を生成し、多くの場合、補助的な可視化を伴います。このケースのように主要なインターフェースがチャットベースである場合、これらの反復的な探索が大量のデータをスキャンして関連する洞察を抽出する一連のクエリをトリガーするため、パフォーマンスが重要になります。

いくつかの特性により、リアルタイムデータベースはこのようなワークロードに特に適しています。例えば、リアルタイム分析データベースは準リアルタイムデータを扱うように設計されており、新しいデータが到着するとほぼ即座に処理して洞察を提供できます。これはAIエージェントにとって極めて重要です。なぜなら、タイムリーで適切な意思決定を行う(または支援する)ために最新の情報が必要となるためです。

コア分析機能も重要です。リアルタイム分析データベースは、大規模なデータセット全体にわたる複雑な集計とパターン検出の実行において優れています。主に生データの保存や取得に焦点を当てた運用データベースとは異なり、これらのシステムは膨大な量の情報を分析するために最適化されています。これにより、トレンドの発見、異常の検出、実用的な洞察の導出が必要なAIエージェントに特に適しています。

リアルタイム分析データベースは、チャットベースのインタラクションや高頻度の探索的ワークロードに不可欠な、インタラクティブなクエリに対する高速パフォーマンスを提供することも期待されています。大量のデータと高いクエリ同時実行性があっても一貫したパフォーマンスを保証し、応答性の高い対話とよりスムーズなユーザー体験を実現します。

最後に、リアルタイム分析データベースは、貴重なドメイン固有のデータを単一の場所に効果的に統合する究極の「データシンク」として機能することがよくあります。異なるソースや形式の重要なデータを同じ傘下に配置することで、これらのデータベースは、AIエージェントが運用システムから切り離されたドメイン情報の統一ビューにアクセスできることを保証します。

<Image img={ml_ai_07} alt='従来のリアルタイム分析' size='md' />

<Image img={ml_ai_08} alt='エージェント向けリアルタイム分析' size='md' />

これらの特性により、リアルタイムデータベースは既に大規模なAIデータ取得ユースケースにおいて重要な役割を果たしています(例:OpenAIによるRocksetの買収)。また、重い計算処理をオフロードしながら、AIエージェントが高速なデータ駆動型の応答を提供することも可能にします。

これにより、リアルタイム分析データベースは、洞察提供に関してAIエージェントにとって好ましい「コンテキストプロバイダー」として位置付けられます。


## 新たなユーザーペルソナとしてのAIエージェント {#ai-agents-as-an-emerging-user-persona}

リアルタイム分析データベースを活用するAIエージェントを理解する上で有効なアプローチは、それらを新しいカテゴリのユーザー、つまりプロダクトマネージャーの言葉で言えば「ユーザーペルソナ」として捉えることです。

<Image img={ml_ai_09} alt='新たなユーザーペルソナとしてのエージェント' size='md' />

データベースの観点から見ると、ユーザーに代わって、あるいは自律的に、調査の実行、反復的な研究と洞察の精緻化、タスクの実行を行うために、潜在的に無制限の数のAIエージェントが同時に大量のクエリを実行することが予想されます。

長年にわたり、リアルタイムデータベースは、システムに直接接続するか、ミドルウェアアプリケーション層を介して接続する人間の対話型ユーザーに適応してきました。典型的なペルソナの例には、データベース管理者、ビジネスアナリスト、データサイエンティスト、データベース上にアプリケーションを構築するソフトウェア開発者などが含まれます。業界は徐々に彼らの利用パターンと要件を学習し、自然な形で、さまざまなユースケースを満たすためのインターフェース、オペレーター、UI、フォーマット、クライアント、およびパフォーマンスを提供してきました。

現在の課題は、AIエージェントのワークロードに対応する準備ができているかということです。これらの利用パターンに対して、どのような特定の機能を再考するか、またはゼロから作成する必要があるのでしょうか。

ClickHouseは、完全な機能を備えたAI体験を提供することを目的とした多数の機能を通じて、これらの課題に対する答えを迅速に提供しています。
