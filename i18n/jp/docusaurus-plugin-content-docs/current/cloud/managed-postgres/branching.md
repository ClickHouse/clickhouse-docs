Managed Postgres では、ポイントインタイム リカバリ (PITR) を使用して、独立したデータベースブランチを作成できます。

ブランチは、既存のデータベースの特定の時点を基に作成される、完全に独立した PostgreSQL デプロイメントです。ブランチは、ソースデータベースに影響を与えることなく、開発、ステージング、テスト、デバッグ、データ検証、リカバリのワークフローに利用できます。

プライマリデータベースとストレージを共有する copy-on-write の実装とは異なり、Managed Postgres のブランチはバックアップから復元され、独立した PostgreSQL デプロイメントとして動作します。

## ブランチの仕組み \{#branching\}

ブランチの作成は、[ポイントインタイムリカバリ (PITR) ](/cloud/managed-postgres/backup-and-restore)で使用されるものと同じ、バックアップおよびリカバリ用のインフラストラクチャを基盤としています。

ブランチを作成すると、Managed Postgres はオブジェクトストレージからベースバックアップを復元し、指定した復旧時点に到達するために必要な WAL セグメントをリプレイして、復元された状態から新しい PostgreSQL デプロイメントをプロビジョニングします。復旧が完了すると、そのブランチはソースデータベースから独立して動作します。

作成されたブランチは、選択した時点におけるソースデータベースの完全なコピーです。

## よくあるユースケース \{#common-use-cases\}

### 開発とテスト \{#dev-and-testing\}

本番環境またはステージング環境のデータベースからブランチを作成し、アプリケーションの変更、移行、新機能を実際のデータに近い環境で検証します。

### ステージング環境 \{#staging-environments\}

本番ワークロードに影響を与えない形で、本番環境に近いステージング環境を維持します。

### データ検証 \{#date-validation\}

本番環境にデプロイする前に、スキーマの変更、索引戦略、クエリ最適化をテストします。

### 復旧と調査 \{#recovery-and-investigation\}

トラブルシューティング、監査、またはアプリケーション動作の検証のために、データベースを特定の時点に復旧します。

## ブランチのサイズ設定 \{#branch-sizing\}

ブランチは独立した PostgreSQL デプロイメントであり、ソースデータベースとは別にサイズを設定できます。

たとえば、本番用のデプロイメントはより大きな構成で運用し、開発用またはステージング用のブランチではコストを抑えるために、より小さなコンピュートプロファイルを使用できます。これにより、チームは本番環境と同等のコンピュートリソースを用意しなくても、一時的な環境を作成できます。

## ブランチの作成時間 \{#branch-creation-time\}

Managed Postgres は NVMe ベースの PostgreSQL ストレージを使用しているため、ブランチはストレージレベルのコピーオンライト機構で作成されるのではなく、バックアップから復元されます。そのため、ブランチは瞬時に作成されるわけではありません。

ブランチの作成時間は通常、数分から数十分程度で、次の要因によって異なります。

* データベースのサイズ
* バックアップのサイズ
* 復旧ポイント
* 再生が必要な WAL の量
* クラスター全体の構成

ほとんどのデプロイメントでは、ブランチは数分以内に利用可能になります。データベースが大規模な場合は、さらに時間がかかることがあります。

ブランチの作成時間がワークフローのボトルネックになっている場合は、ClickHouse チームにお問い合わせください。多くの場合、ワークロードの特性や復旧要件に応じて、ブランチの復旧性能を最適化できます。

## ブランチとローカル開発 \{#branches-v-local-dev\}

よくある質問の 1 つに、すべての開発者が開発環境として本番ブランチを使うべきかというものがあります。

ブランチはテスト、検証、ステージングのワークフローには有用ですが、日常的なアプリケーション開発には、一般に推奨される方法ではありません。各ブランチは独立した PostgreSQL デプロイメントであり、バックアップから復元したうえで、それぞれ個別に維持管理する必要があります。ブランチを大量に作成すると、インフラストラクチャのコストや運用の複雑さが増すおそれがあります。

ほとんどの組織では、次の運用を推奨します。

* ステージング、テスト、デバッグ、検証のワークフローには PostgreSQL のブランチを使用する。
* 日常的な開発にはローカルの PostgreSQL 環境を使用する。
* 必要に応じて、開発用の合成データセットを生成するか、サニタイズ済みのデータセットを使用する。
* 本番環境由来のブランチに対して日常的な開発を直接行うことは避ける。

このアプローチにより、本番システムへの負荷を軽減し、開発速度を高めるとともに、本番データを適切に保護しやすくなります。

Docker を使用してローカルの PostgreSQL 開発環境を作成する方法については、[ローカル開発環境](/cloud/managed-postgres/local-development) を参照してください。

## 推奨ワークフロー \{#recommended-workflow\}

一般的な流れは次のとおりです。

```text
Production Database
        │
        ├─────────────► Branch
        │                  │
        │                  ├── Staging
        │                  ├── Validation
        │                  ├── Migration Testing
        │                  └── Incident Investigation
        │
        └─────────────► Local Development
                               │
                               ├── Docker PostgreSQL
                               ├── Application Migrations
                               └── Synthetic Test Data
```

ブランチは、本番環境に近いデータベースのコピーが必要な環境に最適です。通常の開発作業では、ローカルの PostgreSQL 環境のほうが、より高速で低コストかつスケーラブルなワークフローに適しています。
