# ClickHouse 向けスキーママイグレーションツール \{#schema-migration-tools-for-clickhouse\}

## スキーマ管理とは？ \{#what-is-schema-management\}

スキーマ管理とは、データベーススキーマにバージョン管理の原則を適用する手法です。一般的には、テーブルやカラム、その関係への変更を追跡・自動化することで、スキーマ更新を環境間で繰り返し実行可能かつ監査可能で、一貫性のあるものにするプロセスを指します。スキーマ管理は、データベース内のデータ構造を新たなユースケースやパフォーマンス最適化のために変更する必要があるときに重要になります。

### なぜ重要なのか？ \{#why-is-it-important\}

スキーマ管理ツールを使うと、アプリケーションのデプロイメントと並行してスキーマ変更を自動化できます。新しいアプリケーションバージョンをデプロイする前提条件として、スキーマ変更が必要になることは一般的です。これらのツールは、ユーザーがあるバージョンのデータベースから別のバージョンへ「移行」することから、「schema migration」ツールや「database migration」ツールとも呼ばれます。

スキーマ管理ツールがない場合、データベース変更は手作業となり、ミスが起こりやすく、チーム間や環境間での調整も困難になります。常に DDL を直接データベースに対して実行することもできますが、これらのツールを使うことで、バージョン管理 (変更をバージョン付きコードとして追跡) 、自動デプロイメント (dev / staging / prod 各環境に一貫して変更を適用) 、ロールバックサポート (問題発生時に変更を元に戻す) 、および監査証跡 (何がいつ変更されたかを記録) を実現できます。特に ClickHouse のように、一部の DDL 変更が高コストまたは元に戻せない場合があるシステムでは、レビュー手順を含む体系的なマイグレーションプロセスを持つことが極めて重要です。

### スキーマ管理手法の種類 \{#types-of-schema-management-approaches\}

スキーマ管理ツールは、一般的に大きく 2 つのカテゴリーに分けられます。

#### 命令型 \{#imperative\}

これらのツールは、状態Aから状態Bへ*どのように*移行させるかを記述した、バージョン管理されたSQLファイルを使用します。`CREATE TABLE`、`ALTER TABLE`、`DROP COLUMN` のような明示的なDDL文をファイルに記述し、ツールがそれらのファイルを順番に実行して、どのファイルが適用済みかを管理します。このカテゴリでは、実行するSQLを細部まで指定します。
**例**: *golang-migrate, Goose, Flyway*

#### 宣言的 \{#declarative\}

これらのツールは、まずユーザーが「望ましい状態 (desired state) 」のスキーマ定義を記述するところから始まります。ツールは現在のデータベースと望ましい状態との違いを検出し、必要なマイグレーションを生成して適用します。このアプローチにより、手作業でマイグレーションを書く量を減らし、スキーマドリフトの発生を抑制できます。このカテゴリでは、どの SQL を実行するかはツール側が厳密に決定します。
**例**: *Atlas, Liquibase*

第3のカテゴリとして、データベーススキーマの変更そのものではなく、スキーマ構造ではなくデータ自体の変換に重点を置くツール群も存在します。
**例**: *dbt*
本ドキュメントでは、データベーススキーマの変更を対象としたツールのみに焦点を当てます。

全体として、あなたとチームの運用スタイルに合致するツールを選ぶことを推奨します。手続き型ツールは、どの DDL が実行されるかを完全に可視化できますが、差分の確認やスキーマドリフトの管理には、利用者による継続的な注意と対応が必要になります。宣言的ツールは、多くの定型的なメンテナンス作業を自動化しスキーマドリフトを防ぐのに有用ですが、自動生成されたプランを ClickHouse に適用する前に必ずレビューし、自動生成プランの背後に予期しない mutation や高コストな書き換え (rewrite) が隠れていないことを確認すべきです。

## ツール選定時に検討すべきポイント \{#what-to-consider-when-choosing-a-tool\}

### あなたのチームはすでにどんなツールを使っていますか？ \{#what-does-your-team-already-use\}

多くの場合、既に慣れ親しんでいるエコシステムのツールを選びがちです。チームが Go を主に使っているのであれば、golang-migrate や Goose は自然に感じられるかもしれません。Java エコシステムで開発している場合は、すでに Flyway や
Liquibase を導入しているかもしれません。インフラチームが Terraform や Infrastructure as Code (IaC) のパターンを使っているのであれば、Atlas の宣言的モデルはしっくりくる可能性があります。チームがすでに知っているものを選ぶことには大きな価値があります — 最良のツールとは、実際に導入され、継続的に使われるツールです。

### 望ましいプロセスは何ですか？ \{#what-is-your-desired-process\}

スキーマ変更が組織内をどのように流れるかを考えてください。次のようなニーズがあるかどうかを検討します。

* シンプルな「SQLを書いて、CI で実行して、完了」というワークフロー (例: Goose、golang-migrate)
* 管理された承認ワークフロー、監査証跡、および RBAC (例: Bytebase、Liquidbase)
* スキーマを宣言的に定義し、ツール側で差分を自動算出してほしい (例: Atlas)

要件とプロセスに合致するツールを選択してください。

## 推奨ツール \{#recommended-tools\}

これらは、成熟度、ClickHouse との互換性、コミュニティでの採用状況、および運用上の適合性に基づき、総合的に ClickHouse ユーザーに推奨しているツールです。

### Atlas \{#atlas\}

[Atlas](https://atlasgo.io/guides/clickhouse) は、宣言的アプローチをとる schema-as-code ツールです。HCL または SQL で目的のスキーマ状態を定義すると、Atlas が現在のデータベースを検査し、差分を計算し、マイグレーションプランを生成して適用します (必要に応じて、適用前にレビューすることもできます) 。

**ClickHouse と相性が良い理由:** Atlas は ClickHouse に対してファーストクラスのサポートを提供しており、テーブル、ビュー、materialized view、PROJECTION、パーティション、UDF を含みます。v0.37 (2025 年 9 月) でクラスターサポートが追加されました。HCL とプレーンな SQL の両方でスキーマ定義が可能です。

**注意点:** Atlas はマイグレーションプランを生成しますが、そのプランの *コスト* までは理解しません。差分が一見単純に見える場合 (例: カラム型の変更) でも、マルチテラバイトのテーブルで高コストなミューテーション処理を引き起こす可能性があります。生成されたプランは、適用前に必ずレビューしてください。

**適しているケース:** Infrastructure as Code (IaC) ワークフローと自動ドリフト検出を求めるチーム

* **タイプ:** 宣言的
* **言語:** Go (単一バイナリ)
* **ライセンス:** オープンコア (コミュニティ版は Apache 2.0。高度な機能向けに有償プランあり)
* **クラスターサポート:** あり

### golang-migrate \{#golang-migrate\}

[golang-migrate](https://github.com/golang-migrate/migrate) は、シンプルで広く利用されているマイグレーションランナーです。バージョン付きの SQL ファイルを up/down ステップ付きで作成し、このツールがそれらを順番に適用し、ClickHouse データベース内の `schema_migrations` テーブルで状態を追跡します。

**ClickHouse と相性が良い理由:** シンプルかつ柔軟です。実行したい ClickHouse の DDL をそのまま記述できます。単一の Go バイナリでランタイム依存関係がなく、CI/CD パイプラインや Docker コンテナに容易に組み込めます。

**注意点:** 1 つのマイグレーションファイルに複数の文が含まれていて、そのうち 1 つが途中で失敗した場合、データベースが不完全かつ不整合な状態のまま残り、手動での対応が必要になる可能性があります。これは「1 ファイルにつき 1 文」という運用ルールに従うことで管理可能です。

**最適な利用シナリオ:** ClickHouse インスタンスに対してどの SQL が実行されるかを、シンプルかつ完全に制御したいチーム。

* **タイプ:** 命令型
* **言語:** Go
* **ライセンス:** オープンソース (MIT、無料)
* **クラスターサポート:** あり

### Goose \{#goose\}

[Goose](https://github.com/pressly/goose) は、golang-migrate と似た思想を持つ、別の Go 製マイグレーションランナーです。バージョン付きの SQL ファイル (または複雑なロジック向けの Go 関数) を書き、Goose がそれらを順番に適用し、ClickHouse 内のバージョン管理用テーブルで状態を追跡します。

**ClickHouse と相性が良い理由:** golang-migrate と同様に、SQL ファースト志向、最小限の設定、使いやすい CLI、扱いやすい CI/CD 連携といった本質的な強みがあります。さらに Goose はマイグレーションを Go 関数として記述することもサポートしており、純粋な SQL では表現しづらい複雑なロジックにも柔軟に対応できます。

**注意点:** スキーマの差分取得や自動生成機能はありません。

**最適な利用シナリオ:** すでに Goose を利用しているチーム、または golang-migrate よりも Goose のマイグレーションファイル規約を好むチーム。

* **タイプ:** 命令型
* **言語:** Go (単一バイナリ)
* **ライセンス:** オープンソース (MIT、無償)
* **クラスタサポート:** なし
