# 词汇表



## 原子性 {#atomicity}

原子性确保事务(一系列数据库操作)被视为单一的、不可分割的单元。这意味着事务中的所有操作要么全部发生,要么全部不发生。原子性事务的一个例子是从一个银行账户向另一个账户转账。如果转账的任一步骤失败,整个事务就会失败,资金将保留在第一个账户中。原子性确保资金既不会丢失也不会凭空产生。


## 块 {#block}

块(Block)是组织数据处理和存储的逻辑单元。每个块包含列式数据,这些数据会被统一处理以提升查询执行性能。通过按块处理数据,ClickHouse 能够高效利用 CPU 核心,最大限度地减少缓存未命中并实现向量化执行。ClickHouse 使用多种压缩算法(如 LZ4、ZSTD 和 Delta)对块中的数据进行压缩。


## 集群 {#cluster}

由多个节点(服务器)组成的集合,协同工作以存储和处理数据。


## CMEK {#cmek}

客户管理的加密密钥（CMEK）允许客户使用自己的密钥管理服务（KMS）密钥来加密 ClickHouse 磁盘数据密钥，从而保护静态数据。


## 字典 {#dictionary}

字典是一种键值对映射结构,用于存储各类引用数据。它是 ClickHouse 的一项强大特性,能够在查询中高效使用字典数据,其性能通常优于使用 `JOIN` 关联引用表的方式。


## 分布式表 {#distributed-table}

ClickHouse 中的分布式表是一种特殊的表类型,它本身不存储数据,而是为跨集群多个服务器的分布式查询处理提供统一视图。


## Granule {#granule}

Granule 是未压缩数据块中的一批行。读取数据时,ClickHouse 以 granule 为单位进行访问,而不是逐行访问,这使得分析工作负载中的数据处理速度更快。默认情况下,一个 granule 包含 8192 行。主索引为每个 granule 保存一个索引条目。


## 增量物化视图 {#incremental-materialized-view}

增量物化视图是 ClickHouse 中一种在数据插入时进行处理和聚合的物化视图类型。当新数据插入源表时,物化视图仅对新插入的数据块执行预定义的 SQL 聚合查询,并将聚合结果写入目标表。


## 轻量级更新 {#lightweight-update}

ClickHouse 中的轻量级更新是一项实验性功能,允许您使用标准 SQL UPDATE 语法更新表中的行。与传统 mutation 操作重写整个列或数据部分不同,轻量级更新会创建仅包含更新列和行的"补丁部分"(patch parts)。这些更新通过应用补丁在 SELECT 查询中立即可见,但物理数据只会在后续合并时才真正更新。


## 物化视图 {#materialized-view}

ClickHouse 中的物化视图是一种机制,可在数据插入源表时自动执行查询,将转换或聚合后的结果存储到独立的目标表中,从而加快查询速度。


## MergeTree {#mergetree}

MergeTree 是 ClickHouse 中的一种表引擎,专为高数据写入速率和大数据量场景而设计。它是 ClickHouse 的核心存储引擎,提供列式存储、自定义分区、稀疏主索引以及后台数据合并等功能特性。


## Mutation {#mutation}

ClickHouse 中的 Mutation 是指修改或删除表中现有数据的操作,通常使用 ALTER TABLE ... UPDATE 或 ALTER TABLE ... DELETE 等命令。Mutation 以异步后台进程的方式实现,会重写受变更影响的整个数据部分(data parts),而不是就地修改行。


## 即时变更 {#on-the-fly-mutation}

ClickHouse 中的即时变更是一种机制,允许在提交变更后立即在后续 SELECT 查询中看到更新或删除的结果,而无需等待后台变更过程完成。


## 数据部分 {#parts}

存储表中部分数据的磁盘物理文件。这与分区不同,分区是通过分区键对表数据进行的逻辑划分。


## 分区键 {#partitioning-key}

ClickHouse 中的分区键是在创建表时通过 PARTITION BY 子句定义的 SQL 表达式。它决定了数据如何在磁盘上逻辑分组为分区。分区键的每个唯一值都会形成独立的物理分区,从而实现高效的数据管理操作,例如删除、移动或归档整个分区。


## 主键 {#primary-key}

在 ClickHouse 中,主键决定数据在磁盘上的存储顺序,并用于构建稀疏索引来加速查询过滤。与传统数据库不同,ClickHouse 的主键不强制唯一性——允许多行具有相同的主键值。


## 投影 {#projection}

ClickHouse 中的投影是一个隐藏且自动维护的表,它以不同的顺序存储数据或包含预计算的聚合结果,用于加速查询,尤其是针对主键以外列进行过滤的查询。


## 可刷新物化视图 {#refreshable-materialized-view}

可刷新物化视图是一种物化视图,它会定期在完整数据集上重新执行查询,并将结果存储到目标表中。与增量物化视图不同,可刷新物化视图按计划进行更新,可以无限制地支持复杂查询,包括 JOIN 和 UNION 操作。


## 副本 {#replica}

存储在 ClickHouse 数据库中的数据副本。您可以为相同数据创建任意数量的副本,以实现冗余和可靠性。副本与 ReplicatedMergeTree 表引擎配合使用,使 ClickHouse 能够在不同服务器之间保持多个数据副本同步。


## Shard {#shard}

数据的子集。ClickHouse 始终至少为您的数据保留一个分片。如果不将数据分割到多个服务器,数据将存储在单个分片中。当单台服务器容量不足时,可以将数据分片到多个服务器以分担负载。


## 跳过索引 {#skipping-index}

跳过索引用于在多个连续颗粒的级别存储少量元数据,使 ClickHouse 能够跳过扫描不相关的行。跳过索引提供了一种比投影更轻量的替代方案。


## 排序键 {#sorting-key}

在 ClickHouse 中,排序键定义了数据行在磁盘上的物理存储顺序。如果未指定主键,ClickHouse 会将排序键用作主键。如果同时指定了主键和排序键,则主键必须是排序键的前缀。


## 稀疏索引 {#sparse-index}

一种索引类型,主索引为一组行包含一个条目,而非为单行包含一个条目。对应一组行的条目称为标记(mark)。使用稀疏索引时,ClickHouse 首先识别可能匹配查询的行组,然后分别处理它们以查找匹配项。因此,主索引足够小,可以完全加载到内存中。


## 表引擎 {#table-engine}

ClickHouse 中的表引擎决定了数据如何写入、存储和访问。MergeTree 是最常用的表引擎,它支持快速插入大量数据,这些数据会在后台进行处理。


## TTL {#ttl}

Time To Live (TTL) 是 ClickHouse 的一项功能,可在指定时间段后自动移动、删除或汇总列或行数据。通过删除、移动或归档不再需要频繁访问的数据,您可以更高效地管理存储空间。
