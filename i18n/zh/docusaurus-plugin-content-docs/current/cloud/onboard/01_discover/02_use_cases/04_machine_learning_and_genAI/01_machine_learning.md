## 机器学习数据层 {#machine-learning-data-layer}

您可能听说过这样一个传言:机器学习从业者 80% 的时间都花在清理数据上。
无论这个传言是否属实,不变的事实是,从始至终,数据都是机器学习问题的核心。
无论您是在构建 RAG 管道、微调模型、训练自己的模型,还是评估模型性能,数据都是每个问题的根本。

数据管理可能很棘手,因此该领域涌现出大量工具,这些工具旨在通过解决机器学习数据问题的特定部分来提高生产力。
通常,这些工具表现为在更通用的解决方案之上添加一层抽象,并提供一个固定的接口,从表面上看,这使得将其应用于手头的特定子问题变得更容易。
实际上,这是以牺牲通用解决方案的灵活性为代价,来换取特定任务的易用性和简单性。

<Image img={machine_learning_data_layer} size='sm' />

这种方法存在几个缺点。
与通用解决方案加上支持应用代码相比,一系列级联的专用工具、产品和服务会带来不必要的架构复杂性和数据成本增加的风险。
很容易在不知不觉中发现自己拥有一长串工具和服务,每个工具仅用于单个步骤。

这些风险通常体现在两个方面:

1. **学习、维护和切换成本**

机器学习架构可能会因各种工具和组件而变得杂乱无章,从而形成一个碎片化且难以学习和管理的环境,增加了故障点和成本蔓延。

2. **数据重复和传输成本**

在机器学习管道中使用多个离散但重叠的数据系统可能会引入不必要且通常代价高昂的数据传输开销。

向量数据库就是这种权衡的一个很好的例子。
向量数据库专为存储和搜索向量这一高度特定的机器学习任务而设计。
虽然这在某些架构中可能是正确的选择,但在其他架构中,向量数据库可能是技术栈中不必要的新增内容,因为它是又一个需要集成、管理以及进行数据传输的系统。
大多数现代通用数据库都开箱即用地支持向量(或通过插件),并具有更广泛和跨领域的能力。
换句话说,在这些架构中可能根本不需要专门用于处理向量的全新数据库。
关键在于向量特定的便利功能(例如内置嵌入模型)是否是关键任务且值得付出成本。

### 数据探索 {#data-exploration}

在定义机器学习问题、目标和成功标准之后,常见的第一步是探索将用于模型训练和评估的相关数据。

在此步骤中,需要分析数据以了解其特征、分布和关系。
这个评估和理解的过程是迭代的,通常会导致跨数据集执行一系列临时查询,其中查询响应速度至关重要(以及成本效益和准确性等其他因素)。
随着公司存储越来越多的数据用于机器学习目的,检查现有数据的问题变得更加困难。

这是因为在传统数据系统中,分析和评估查询在大规模场景下往往会变得缓慢得令人难以忍受或无法接受。
一些大型厂商通过大幅提高成本来降低查询时间,并通过按查询收费或按扫描字节数收费的方式阻止临时评估。
工程师可能不得不将数据子集拉取到本地机器上,作为对这些限制的妥协。

另一方面,ClickHouse 是一个实时数据仓库,因此用户可以从行业领先的分析计算查询速度中受益。
此外,ClickHouse 从一开始就提供高性能,不会将关键的查询加速功能限制在更高的定价层级之后。
ClickHouse 还可以直接从对象存储或数据湖查询数据,支持 Iceberg、Delta Lake 和 Hudi 等常见格式。
这意味着无论您的数据存储在何处,ClickHouse 都可以作为机器学习工作负载的统一访问和计算层。

ClickHouse 还拥有一套广泛的预构建统计和聚合函数,可扩展到 PB 级数据,使编写和维护执行复杂计算的简单 SQL 变得容易。
凭借对最精细精度数据类型和编解码器的支持,您无需担心降低数据的粒度。


虽然用户可以在 ClickHouse 中直接转换数据，或在插入前通过 SQL 查询进行转换，ClickHouse 还可以通过 [chDB](/chdb) 在 Python 等编程环境中使用。
这使得嵌入式 ClickHouse 可以作为一个 Python 模块对外提供，在笔记本环境中用于转换和处理大型数据帧。
因此，数据工程师可以在客户端侧执行转换工作，并将结果物化为集中式 ClickHouse 实例中的特征表。

### 数据准备与特征提取 {#data-preparation-and-feature-extraction}

随后对数据进行准备：包括清洗、转换，并用于提取将用于训练和评估模型的特征。
这一组件有时被称为特征生成或特征提取流水线，是机器学习数据层中的另一部分，通常会在这里引入新的工具。
像 Neptune 和 Hopsworks 这样的 MLOps 产品，就是用于编排此类流水线的众多数据转换工具中的代表。
然而，由于它们与所操作的数据库是相互独立的工具，这些系统可能比较脆弱，一旦出现中断，往往需要人工干预和修复。

相比之下，通过 [物化视图](/materialized-views)，可以在 ClickHouse 中直接、轻松地完成数据转换。
物化视图会在新数据插入 ClickHouse 源表时自动触发，用于在数据到达时轻松地抽取、转换和修改数据，从而无需自行构建和监控定制流水线。
当这些转换需要在可能无法全部装入内存的完整数据集上进行聚合时，借助 ClickHouse，可以避免为了在本地机器的数据帧上完成这一步而进行各种勉强适配。
对于那些在本地评估更为方便的数据集，[ClickHouse local](/operations/utilities/clickhouse-local) 以及 [chDB](/chdb) 是很好的选择，它们允许用户将 ClickHouse 与 Pandas 等标准 Python 数据库库结合使用。

### 训练与评估 {#training-and-evaluation}

在这一阶段，特征会被划分为训练集、验证集和测试集。
这些数据集会进行版本化管理，然后被各自对应的阶段使用。

在流水线的这一阶段，通常会在机器学习数据层引入另一类专用工具——特征存储。
特征存储通常是构建在数据库之上的一层抽象，它提供一组专门用于管理模型训练、推理和评估数据的便捷功能。
这些便捷功能的例子包括版本管理、访问控制，以及将特征定义自动转换为 SQL 语句。

在特征存储场景中，ClickHouse 可以充当：

**数据源** - ClickHouse 能够查询或摄取 70 多种不同的文件格式，包括 Iceberg 和 Delta Lake 等数据湖格式，因此非常适合作为长期的数据存储和查询系统。
通过使用对象存储实现存储与计算的分离，ClickHouse Cloud 还能让数据被长期保存，而计算资源则可以缩容甚至完全空闲，以尽量降低成本。
灵活的编码格式结合列式存储以及磁盘上的数据排序，可以最大化压缩比，从而将存储需求降到最低。
借助内置函数，用户可以轻松将 ClickHouse 与数据湖集成，对对象存储中的数据进行就地查询。

**转换引擎** - SQL 为声明数据转换提供了一种自然的方式。
结合 ClickHouse 的分析和统计函数，这些转换可以变得简洁且高度优化。
除了作用于 ClickHouse 表（在 ClickHouse 用作数据存储的场景下）之外，表函数还允许对存储为 Parquet 等格式、位于本地磁盘或对象存储中的数据，乃至其他数据存储系统（如 Postgres 和 MySQL）执行 SQL 查询。
完全并行化的查询执行引擎结合列式存储格式，使 ClickHouse 能够在数秒内对 PB 级数据完成聚合运算；与在内存数据帧上进行转换不同，用户不会受到内存容量的限制。
此外，物化视图支持在插入时对数据进行转换，从而将计算负载从查询时转移到数据加载时。
这些视图可以利用同样丰富的分析和统计函数，非常适合用于数据分析和汇总。
如果 ClickHouse 现有的分析函数无法满足需求，或需要集成自定义库，用户还可以使用用户自定义函数（UDF）。

#### 离线特征存储 {#offline-feature-store}

离线特征存储用于模型训练。
这通常意味着特征本身是通过批处理的数据转换流水线生成的（如上文所述），并且对这些特征的可用性一般没有严格的延迟要求。


ClickHouse 能够从多个数据源读取数据并通过 SQL 查询进行转换,这些查询的结果也可以通过 `INSERT INTO SELECT` 语句持久化到 ClickHouse 中。
由于转换通常按实体 ID 分组并返回多列结果,ClickHouse 的模式推断功能可以自动检测这些结果所需的类型,并生成相应的表模式来存储它们。
生成随机数和统计采样的函数允许以每秒数百万行的速度高效地迭代和扩展数据,以供模型训练管道使用。

通常,特征在表中以时间戳的形式表示,指示实体和特征在特定时间点的值。
如前所述,训练管道通常需要获取特定时间点和分组的特征状态。ClickHouse 的稀疏索引支持快速过滤数据,以满足时间点查询和特征选择过滤的需求。
虽然 Spark、Redshift 和 BigQuery 等其他技术依赖于缓慢的有状态窗口方法来识别特定时间点的特征状态,但 ClickHouse 支持 ASOF(截至此时间)LEFT JOIN 查询和 argMax 函数。
除了简化语法外,这种方法通过使用排序和合并算法在大型数据集上实现了高性能。
这使得特征组可以快速构建,从而减少训练前的数据准备时间。

#### 在线特征存储 {#online-feature-store}

在线特征存储用于存储推理所需的最新版本特征,并实时应用。
这意味着这些特征需要以最小延迟进行计算,因为它们是实时机器学习服务的一部分。

<Image img={online_feature_store} size='sm' />

作为实时分析数据库,ClickHouse 可以以低延迟处理高并发查询工作负载。
虽然这需要数据通常采用非规范化形式,但这与训练和推理时使用的特征组存储方式相一致。
重要的是,得益于其日志结构合并树,ClickHouse 能够在承受高写入工作负载的同时提供出色的查询性能。
在线存储需要这些特性来保持特征的实时更新。
由于特征已经在离线存储中可用,它们可以通过现有功能(例如 `remoteSecure`)轻松物化到同一 ClickHouse 集群或不同实例中的新表。
通过 exactly-once 的 Kafka Connect 或 ClickHouse Cloud 中的 [ClickPipes](/integrations/clickpipes/kafka) 与 Kafka 的集成,也使得从流式数据源消费流式数据变得简单可靠。

许多现代系统需要同时使用离线和在线存储,很容易得出需要两个专门的特征存储的结论。
然而,这引入了保持这两个存储同步的额外复杂性,当然也包括在它们之间复制数据的成本。

像 ClickHouse 这样的实时数据仓库是一个可以同时支持离线和在线特征管理的单一系统。
ClickHouse 高效处理流式和历史数据,并具有在为实时推理和离线训练提供特征时所需的无限扩展能力、性能和并发能力。

在考虑在此阶段使用特征存储产品与直接利用实时数据仓库之间的权衡时,值得强调的是,版本控制等便利功能可以通过表或模式设计等传统数据库范式来实现。
其他功能,例如将特征定义转换为 SQL 语句,作为应用程序或业务逻辑的一部分可能提供更大的灵活性,而不是存在于一个固定的抽象层中。

### 推理 {#inference}

模型推理是运行训练好的模型以获取输出的过程。
当推理由数据库操作触发时(例如插入新记录或查询记录),推理步骤可以通过定制作业或应用程序代码来管理。

另一方面,它也可以在数据层本身进行管理。ClickHouse [用户定义函数(UDF)](/sql-reference/functions/udf) 使用户能够在插入或查询时直接从 ClickHouse 调用模型。
这提供了将传入数据传递给模型、接收输出并自动将这些结果与摄取的数据一起存储的能力——所有这些都无需启动其他进程或作业。
这还提供了一个单一接口 SQL 来管理此步骤。

### 向量存储 {#vector-store}

向量存储是一种专门用于存储和检索向量的特定类型数据库,通常是数据片段(如文本或图像)的嵌入表示,以数值方式捕获其底层含义。
向量是当今生成式 AI 浪潮的核心,并在无数应用中使用。


向量数据库的核心操作是"相似性搜索",即根据数学度量标准找到彼此"最接近"的向量。
向量数据库之所以流行,是因为它们采用了专门的策略来加速向量比较操作。
这些技术通常采用近似计算方式进行向量比较,而不是将输入向量与所有存储的向量逐一比较。

这类新工具面临的问题是,许多通用数据库(包括 ClickHouse)都提供开箱即用的向量支持,并且通常内置了这些近似算法的实现。
ClickHouse 专为高性能大规模分析而设计,能够高效执行精确的向量比较。
这意味着您可以获得精确结果,而无需依赖近似计算,同时不会牺牲性能。

### 可观测性 {#observability}

机器学习应用上线后,会生成包括日志和追踪数据在内的各类数据,这些数据能够为模型行为、性能表现以及潜在改进方向提供有价值的洞察。

基于 SQL 的可观测性是 ClickHouse 的另一个关键应用场景,ClickHouse 的成本效益比替代方案高出 10-100 倍。
事实上,许多可观测性产品本身就是基于 ClickHouse 构建的。
凭借业界领先的数据摄取速率和压缩比,ClickHouse 能够以高性价比和极快的速度支撑任意规模的机器学习可观测性需求。
